最近看到一篇文章《[Claude Code told me what tools it needs to work faster. Oh boy I was missing so many things.](https://sderosiaux.substack.com/p/claude-code-told-me-what-tools-it)》，该文章通过一次让 Claude Code 自查本机环境的实验，整理出 ripgrep、fd、fzf、DuckDB、git-delta、xh、watchexec、just、semgrep 等能显著提升 AI 编码助手效率的工具。

作者借此提出：提升 AI 编码效果，不只是改 prompt，更要把 AI 当成需要专用开发环境的协作者。

于是，出于好奇，我分别问 Codex 和 Claude 下面的问题：

> ***“为了有效地帮助我解决问题，您还缺少哪些工具？请分析已安装的软件、缺失的软件、损坏的软件和冗余的软件。并根据对您帮助我的能力的影响程度进行优先级排序。”***

### Codex

我让 Codex 整理出了一份报告。其中，Codex 给出了缺失的软件：

P0 优先级：

- `gh`

  - **状态**: 缺失

  - **影响**: 高

  - **原因**: 会直接影响 GitHub 相关任务，例如 PR、issue、release、workflow、仓库元信息查询。


- `psql`

  - **状态**: 命令缺失

  - **影响**: 高

  - **原因**: PostgreSQL 客户端不可用，会影响数据库连接、SQL 验证和故障排查。


- `gradle`

  - **状态**: 命令不可用

  - **影响**: 高

  - **原因**: Gradle 项目的构建、测试、依赖检查会受影响。


P1 优先级：

- `redis-cli`

  - **状态**: 缺失

  - **影响**: 中

  - **原因**: 影响 Redis 相关排查和验证。


- `mysql`

  - **状态**: 缺失

  - **影响**: 中

  - **原因**: 影响 MySQL 项目排查和连接。


- `poetry`

  - **状态**: 缺失

  - **影响**: 中

  - **原因**: Python 项目如果使用 Poetry，会降低兼容性和排查效率。


- `deno`

  - **状态**: 缺失

  - **影响**: 中低

  - **原因**: 只在 Deno 项目中会明显受影响。


- `playwright`

  - **状态**: CLI 缺失

  - **影响**: 中低

  - **原因**: 本地脚本化浏览器测试少一个通道，但当前已有其他浏览器自动化能力可用。

我在本地开发基本上都是通过 docker 容器进行，所以不会安装数据库、缓存相关的工具。gradle 不常用，所以没有安装。唯一可以安装的是 gh。

按照 Codex 的检查报告，我删除了一些冗余版本的软件。

Claude 帮我安装了这些缺失的软件，并做了一个总结：

1. **已安装**

文件搜索与导航

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `fd` | `find` 的现代替代品 | 快速查找文件，AI 更少出错 |
| `ripgrep (rg)` | 快速代码搜索 | AI 搜索代码模式更高效 |
| `fzf` | 模糊搜索 | AI 可建议历史命令/文件 |

输出增强

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `bat` | `cat` 的替代品，带语法高亮 | AI 读取文件更清晰 |
| `git-delta` | Git diff 可视化 | AI 理解代码变更更直观 |
| `eza` | `ls` 的现代版 | 文件列表更结构化 |

数据处理

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `jq` | JSON 处理 | AI 解析 JSON API 输出 |
| `yq` | YAML 处理（jq 的 YAML 版） | AI 处理 Kubernetes/配置文件 |
| `duckdb` | 嵌入式 SQL 数据库 | AI 用 SQL 分析数据，无需写 Python |

HTTP 与 API

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `xh` | `httpie` 的现代版 | AI 测试 API，输出更结构化 |
| `httpie` | HTTP 客户端 | AI 调试 API 端点 |

代码质量

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `semgrep` | 静态代码分析 | AI 辅助代码审查，查找漏洞 |

自动化

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `just` | 现代 `make` | AI 管理项目命令更简洁 |
| `watchexec` | 文件变化自动执行 | AI 监控文件变化自动构建 |

文本处理

| 工具 | 用途 | AI 受益场景 |
|------|------|-------------|
| `sd` | 交互式 `sed` | AI 批量替换更安全 |


2. **推荐安装** 

| 工具 | 用途 | 安装命令 |
|------|------|----------|
| `tldr` | 简化版 man 手册 | `brew install tldr` |
| `procs` | `ps` 的现代版 | `brew install procs` |
| `bottom` | `top` 的现代版 | `brew install bottom` |

3. **Shell 集成配置**

fzf 集成

```bash
# ~/.zshrc
[ -f ~/.fzf.zsh ] && source ~/.fzf.zsh
```

运行安装脚本：
```bash
/opt/homebrew/opt/fzf/install
```

zsh 自动补全

```bash
# ~/.zshrc
export FPATH="/opt/homebrew/share/zsh/site-functions:$FPATH"
autoload -Uz compinit && compinit
```

eza 别名

```bash
# ~/.zshrc
alias ls='eza'
alias ll='eza -la'
alias tree='eza --tree'
```

bat 集成

```bash
# ~/.zshrc
alias cat='bat'
```

just 补全

```bash
# ~/.zshrc
source (just --completions zsh | psub)  # fish
# zsh:
eval "$(just --completions zsh)"
```

4. **项目级配置**

justfile 示例

```just
# Justfile for project

test:
    echo "Running tests..."
    cargo test

build:
    echo "Building..."
    cargo build

dev:
    watchexec -e rs,tml cargo run
```

> 我之前是使用 go-task 来管理 taskfile，现在可以删除了，改用 just 来管理 justfile。

.semgrep.yml 示例

```yaml
rules:
  - id: possible-sql-injection
    patterns:
      - pattern: execute($QUERY)
    message: "Possible SQL injection"
    languages: [python]
    severity: ERROR
```
