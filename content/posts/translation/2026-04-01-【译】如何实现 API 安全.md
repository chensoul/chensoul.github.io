大多数投入生产的 API 都有一定的安全措施。大多数时候，启用了 HTTPS，需要 API 密钥，甚至在部署之前甚至还有一个快速的代码审查。

按照大多数标准，框已勾选。但是，勾选框和安全的 API 不是一回事。一个常见且代价高昂的例子是：一个 API 在每次请求时正确验证凭据，但从不检查这些凭据是否授予对正在访问的特定资源的访问权限。换句话说，身份验证有效，但没有适当的授权。

这样的 API 不能被称为安全的，但有时在快乐路径中没有什么能揭示这个问题，直到有人发现这个漏洞。

这正是使 API 安全真正棘手的原因。策略可能有据可查，但了解何时使用特定策略可能会令人困惑。在这篇文章中，我们将看看各种 API 安全策略，并尝试理解哪种策略适用于哪种场景。

## API 安全策略

![API 安全策略概览](01.webp)

### 1. 身份验证（Authentication）

**验证用户是谁**

- **API 密钥**：简单但有限，适用于服务到服务的通信
- **OAuth 2.0**：行业标准，适用于第三方授权
- **JWT**：无状态，适用于微服务
- **Session**：有状态，适用于 Web 应用

### 2. 授权（Authorization）

**验证用户可以做什么**

- **RBAC（基于角色的访问控制）**：根据用户角色分配权限
- **ABAC（基于属性的访问控制）**：根据用户属性、资源属性、环境条件
- **权限检查**：在资源级别验证访问权限

### 3. 输入验证

**防止恶意输入**

- **类型检查**：验证数据类型
- **长度限制**：防止缓冲区溢出
- **白名单**：只允许已知安全的值
- **SQL 注入防护**：参数化查询
- **XSS 防护**：转义输出

### 4. 速率限制

**防止滥用**

- **每用户限制**：基于用户 ID
- **每 IP 限制**：基于 IP 地址
- **全局限制**：保护整体系统
- **令牌桶算法**：允许突发流量
- **滑动窗口**：更平滑的限制

### 5. 加密

**保护数据传输和存储**

- **HTTPS/TLS**：传输中加密
- **数据加密**：静态数据加密
- **密钥管理**：安全存储和管理密钥

### 6. 审计和监控

**检测和响应威胁**

- **日志记录**：记录所有 API 请求
- **异常检测**：识别可疑模式
- **告警**：实时通知安全事件
- **合规性报告**：满足监管要求

## 何时使用哪种策略

| 场景 | 推荐策略 |
|------|----------|
| 公共 API | OAuth 2.0 + 速率限制 + 输入验证 |
| 内部微服务 | JWT + mTLS + 授权检查 |
| 移动应用 | OAuth 2.0 + JWT + 设备指纹 |
| Web 应用 | Session + CSRF 保护 + 输入验证 |
| 服务到服务 | API 密钥 + mTLS + 速率限制 |

## 常见安全漏洞

1. **身份验证有效但授权缺失**
   - 问题：验证了用户身份，但没有检查是否有权访问资源
   - 解决：在每个资源端点实施权限检查

2. **过度依赖客户端验证**
   - 问题：客户端验证可以被绕过
   - 解决：始终在服务器端验证

3. **不安全的直接对象引用（IDOR）**
   - 问题：`/api/users/123` 可以被任何人访问
   - 解决：验证请求者是否有权访问该资源

4. **缺乏速率限制**
   - 问题：API 可以被滥用或 DDoS 攻击
   - 解决：实施多层速率限制

5. **日志中记录敏感数据**
   - 问题：密码、令牌等敏感信息被记录
   - 解决：过滤敏感字段，使用安全的日志实践

## 译者总结

这篇文章强调了 API 安全的关键洞察：

**核心问题**：
- 身份验证 ≠ 授权
- 勾选安全清单 ≠ 真正安全
- 需要理解何时使用哪种策略

**关键策略**：
1. **身份验证**：验证用户是谁（API 密钥、OAuth、JWT、Session）
2. **授权**：验证用户可以做什么（RBAC、ABAC、权限检查）
3. **输入验证**：防止恶意输入（类型检查、长度限制、白名单）
4. **速率限制**：防止滥用（每用户、每 IP、全局）
5. **加密**：保护数据（HTTPS、数据加密、密钥管理）
6. **审计和监控**：检测威胁（日志、异常检测、告警）

**常见漏洞**：
- 身份验证有效但授权缺失（最常见）
- 过度依赖客户端验证
- 不安全的直接对象引用（IDOR）
- 缺乏速率限制
- 日志中记录敏感数据

**最佳实践**：
- 始终在服务器端验证
- 在每个资源端点检查权限
- 实施多层速率限制
- 过滤日志中的敏感数据
- 定期审查和更新安全策略

API 安全是一个多层次的问题，需要综合使用多种策略来保护 API 免受各种威胁。
