### 3. 给 N8n 获取 keep 数据添加 AI 分析

![20260315-week-review-02](20260315-week-review/02.webp)

系统提示词：

```markdown
你是一位专业的跑步数据分析师，面向大众跑者与进阶跑者。请根据用户提供的跑步数据，用 2–4 句中文给出评价和可执行的训练建议。在数据缺失时不臆测、不展开科普。
```

用户提示词：

```markdown
请分析以下跑步数据，给出专业分析。\n\n时间：2026-03-12 09:00:44\n距离：3.01 公里\n用时：21 分 34 秒\n配速：7:09 分钟/公里\n运动消耗：262千卡\n运动负荷：36\n平均心率：141\n最大心率：167\n设备：OPPO Watch 46mm
```

最后发送到电报的内容：

```markdown
🏃‍♀️ 2026-03-12 09:00:44，使用 OPPO Watch 46mm 户外跑步 3.01 公里，用时 21 分 34 秒，运动负荷 36，配速 7:09 分钟/公里，平均心率 141，最大心率 167。 #running

🤖 AI 分析：
本次 3.01 公里、配速 7:09/公里，运动负荷 36，属于中等强度的有氧跑；平均心率 141、最大 167，说明大部分时间在有氧区，但存在短时心率冲高。建议下次若以打有氧基础为主，可略降配速（如 7:20–7:30/公里），把平均心率压在 135 左右，跑完仍能轻松说话；若保持当前强度，每周可安排 2–3 次类似跑量，并搭配 1 次休息或轻松跑，便于恢复与进步。
```

### 4. 每日简报

看完 [我让龙虾替我工作了 38 天，它做了什么？](https://atbug.com/openclaw-38-days-what-did-it-do/) 之后，我顺手研究了一下作者开发的 [openclaw skills](https://github.com/addozhang/openclaw-forge)。文章里复盘了三个 agent 的产出，结构很清晰：

#### Nova（数字管家）

- 维护开源项目、回复 issue
- 从零到发布完成 Nexus MCP Server
- 开发了一批 skill：daily-briefing、google-tasks、tech-doc-translator、research-prep、quick-capture
- 批量提交 44 篇博客到微软 MVP Portal

#### Forge（编程搭档）

- 重写 Android 断食追踪 App（Kotlin + Jetpack Compose）
- 开发 Feed Pulse（Telegram RSS 订阅机器人）
- 参与技术决策和可行性分析

#### Assistant（打杂）

- 协助 GitHub Copilot Dev Days 广州筹备
- 帮作者太太整理 HR 数据

其中有一个 `daily-briefing` agent，会在每天早 7 点自动推送天气、待办、Hacker News Top 20 到 Telegram。

这和我在 GitHub Action 里用 Python 写的 [每日简报](https://github.com/chensoul/chensoul/blob/main/daily_review.py) 很像，于是我顺手改了自己的脚本。现在这份每日简报会发送日期、天气（wttr.in）、今日待办（GitHub Issues）、今日指数/诗词/名言/Trending、WakaTime、跑步距离、昨日收藏（Linkding）、Hacker News 等信息。

待办这一块我没有用 Google Tasks。最开始考虑过 memos 的 todo，后来放弃，最后改成直接用 GitHub Issues。后面如果遇到更合适、API 更顺手的待办工具，可能还会继续换。

**为什么不使用 openclaw 来实现每日简报？**

像每日简报这种输入和输出都比较确定的任务，用 GitHub Action 就够了，也不需要 AI 参与。直接上 openclaw 反而会增加 token 消耗，没必要。
