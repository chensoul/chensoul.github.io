## 公司如何发布代码到生产

以下是从规划到生产的 11 个步骤：

1. **产品负责人启动整个过程**
   在 Jira 等工具中创建用户故事。

2. **开发团队执行冲刺计划活动**
   将用户故事添加到冲刺。

3. **开发者处理分配的故事**
   一旦故事完成，他们提交代码到 Git 并推送到 GitHub。

4. **Jenkins 构建并运行代码**
   通过测试和质量检查工具，如 JUnit、Jacoco 和 SonarQube。

5. **如果构建成功**
   它存储在 artifactory（如 JFrog）中。Jenkins 还通过 Docker 将构建部署到开发环境。

6. **接下来，功能部署到 QA 环境**
   由于多个团队可能在同一代码库上工作，将创建多个 QA 环境。

7. **QA 团队使用特定 QA 环境**
   运行多种测试类型，如 QA、回归和性能。

8. **一旦 QA 验证完成**
   功能部署到 UAT（用户验收测试）环境。

9. **UAT 测试验证功能是否满足用户需求**

10. **一旦 UAT 测试成功**
    构建成为发布候选。它们基于特定时间表部署到生产环境。

11. **SRE 团队使用工具监控生产环境**
    如 ELK 和 Prometheus，并在出现问题时处理警报。

## 什么是事件溯源？

事件溯源范式用于设计具有确定性的系统。这改变了正常系统设计的理念。

### 工作原理

事件溯源设计不是将订单状态记录在数据库中，而是将导致状态变化的事件持久化在事件存储中。事件存储是仅追加日志。事件必须用递增数字排序以保证它们的顺序。订单状态可以从事件重建并维护在 OrderView 中。如果 OrderView 宕机，我们总是可以依赖作为真相源的事件存储来恢复订单状态。

### 详细步骤

#### 非事件溯源
- **步骤 1 和 2**：Bob 想购买产品。订单创建并插入数据库。
- **步骤 3 和 4**：Bob 想将数量从 5 改为 6。订单用新状态修改。

#### 事件溯源
- **步骤 1 和 2**：Bob 想购买产品。创建 NewOrderEvent，排序，并存储在事件存储中，eventID=321。
- **步骤 3 和 4**：Bob 想将数量从 5 改为 6。创建 ModifyOrderEvent，排序，并持久化在事件存储中，eventID=322。
- **步骤 5**：订单视图从订单事件重建，显示订单最新状态。

## 数据湖架构如何工作

数据湖是一个集中存储系统，以原始形式保存大量结构化、半结构化和非结构化数据。它允许组织大规模存储数据，并稍后处理以满足不同需求。

### 工作原理（高层面）

1. **数据从多样来源收集以进行摄入**
   如 S3、Excel 文件、数据库、视频、图像、传感器和 PDF。

2. **摄入通过实时流或批量/计划作业发生**
   取决于数据类型和来源。

3. **摄入的数据首先存储在原始数据存储中**
   在任何转换或处理发生之前。

4. **数据然后通过批量或实时处理系统处理**
   以清理、转换和准备使用。

5. **处理的数据存储并准备好供下游消费**
   由各种工具和平台。

6. **最终数据消费通过仪表板、AI 模型、数据仓库发生**
   实时警报和报告系统。

## Netflix 如何构建分布式计数器

分布式计数器是一个系统，其中计数事件的责任分布在网络中的多个服务器或节点上。Netflix 需要跟踪和测量多个用户交互以做出实时决策和优化基础设施。

为此，他们构建了分布式计数器抽象。

### Netflix 的分布式计数器抽象在四个主要层中操作

确保高性能、可扩展性和最终一致性。

1. **客户端 API 层**
   用户通过与系统发送 AddCount、GetCount 或 ClearCount 请求进行交互。Netflix 数据网关高效处理和路由这些请求。

2. **事件日志和 TimeSeries 存储**
   事件存储在 Netflix TimeSeries 抽象中以获得可扩展性。每个事件都标记有事件 ID 以确保幂等性。为避免数据库竞争，事件分组到称为桶的时间分区中。数据存储在 Cassandra 中。

3. **Rollup 管道或聚合**
   Rollup 队列收集事件更改并批量处理它们。聚合在不可变时间窗口中发生，确保准确的 rollup 计算。数据存储在 Cassandra Rollup Store 中以实现最终一致性。

4. **读取优化（缓存和查询处理）**
   聚合计数器值缓存在 EVCache 中以实现超快读取。如果缓存值过时，后台 rollup 刷新更新它。此模型允许 Netflix 每秒处理 7.5 万次请求，具有个位数毫秒延迟。

## TCP 握手如何工作

握手过程有三个步骤：

### 1. SYN（同步）
想象成一方伸出手握手。过程从客户端发送带有 SYN 标志的 TCP 段到服务器开始。同时，初始序列号（ISN）随消息一起发送。

### 2. SYN-ACK
这就像第二方同意握手。服务器接收 SYN 段并响应发送带有 SYN 标志和 ACK 标志的 TCP 段到客户端。ACK 标志旨在通过递增 ISN 来确认收到客户端的 SYN 段。

### 3. ACK
这是双方握手的地方。客户端接收 SYN-ACK 段并通过发送另一个带有 ACK 标志设置的 TCP 段来确认服务器的响应。序列号设置为递增后服务器的 ISN。

### 一些值得考虑的要点

- **SYN 和 ACK 是控制标志**
  用于管理通信过程的不同方面。

- **序列号（ISN）帮助客户端和服务器跟踪消息顺序**
  所以不会遗漏任何东西。

## 译者总结

这篇文章涵盖了事件溯源、CI/CD、数据湖、分布式计数器和 TCP 握手的核心概念：

**发布代码到生产 11 步骤**：
| 步骤 | 操作 | 工具 |
|------|------|------|
| 1 | 创建用户故事 | Jira |
| 2 | 冲刺计划 | - |
| 3 | 开发提交 | Git、GitHub |
| 4 | 构建测试 | Jenkins、JUnit、SonarQube |
| 5 | 存储部署 | JFrog、Docker |
| 6 | QA 环境部署 | - |
| 7 | QA 测试 | - |
| 8 | UAT 部署 | - |
| 9 | UAT 测试 | - |
| 10 | 生产部署 | - |
| 11 | 监控 | ELK、Prometheus |

**事件溯源 vs. 非事件溯源**：
| 特性 | 非事件溯源 | 事件溯源 |
|------|------------|----------|
| 存储 | 状态 | 事件 |
| 更新 | 修改状态 | 追加事件 |
| 恢复 | 困难 | 从事件重建 |
| 真相源 | 数据库 | 事件存储 |

**数据湖 6 步骤**：
1. 数据收集
2. 数据摄入
3. 原始存储
4. 数据处理
5. 存储准备
6. 数据消费

**Netflix 分布式计数器 4 层**：
| 层 | 职责 | 存储 |
|------|------|------|
| 客户端 API | 请求路由 | - |
| 事件日志 | 事件存储 | Cassandra |
| Rollup 管道 | 批量聚合 | Cassandra Rollup |
| 读取优化 | 缓存 | EVCache |

**TCP 握手 3 步骤**：
| 步骤 | 标志 | 说明 |
|------|------|------|
| SYN | SYN | 客户端发起 |
| SYN-ACK | SYN+ACK | 服务器确认 |
| ACK | ACK | 客户端确认 |

**关键洞察**：
- 事件溯源提供确定性和可恢复性
- CI/CD 需要多阶段验证
- 数据湖支持多样数据类型
- Netflix 每秒处理 7.5 万请求
- TCP 握手确保可靠连接

理解这些概念对构建可靠、可扩展系统至关重要。
