我一直在琢磨分布式系统工程师在职场上学到的那套教训。我们很大程度上是靠线上流量里踩坑留下的伤疤来「上课」的。伤疤当然是提醒，但要是能让更多工程师十指齐全，显然更好。

新人往往会自学 [Fallacies of Distributed Computing](http://en.wikipedia.org/wiki/Fallacies_of_Distributed_Computing) 和 [CAP 定理](http://codahale.com/you-cant-sacrifice-partition-tolerance/)。可这些多半还是抽象概念，缺少那种能让缺乏经验的工程师立刻动手的、直接可执行的建议（当然，Rotem-Gal-Oz [对这套谬论的整理](http://www.rgoarchitects.com/Files/fallacies.pdf)也非常好）。奇怪的是，新人起步时常几乎得不到什么情境与脉络。

下面是我作为分布式系统工程师认为值得告诉新人的一些教训。有的微妙，有的出人意料，但都不应存在争议。这份列表是给新上手的分布式系统工程师用来校准思路的，不是大全，却是一个像样的起点。

它最糟的一点，是主要谈技术问题，却很少谈工程师会碰到的「人的问题」。分布式系统要更多机器、更多资本，工程师往往就要跟更多团队、更大组织打交道。对任何软件开发者来说，人的问题通常才是最难的部分——做分布式可能尤其如此。

我们的背景、教育与经历，都很容易把我们推向技术方案，即便社交/流程上的办法更高效、也更让人满意。我们尽量把这层偏向扳回来一点：人比机器好说话，只不过接口没那么标准。

好了，进入正文。

**分布式系统不一样，因为它们经常失败。** 若问什么把分布式系统工程和其他软件工程领域区分开，新人常答「延迟」，以为那才是难点。

其实不对。区分开来的，是**失败的概率**——更糟的是**部分失败**的概率。本机上一个行为良好的互斥锁若 `unlock` 报错，大可以认定进程不稳定然后干脆崩掉；但**分布式**互斥锁的 `unlock` 失败，就必须写进锁协议里统筹处理。

没做过分布式计算的工程师，常会想出「那就同时写两台机器」或「失败就一直重试写」这类主意。他们还没真正把下面这点吃进心里（尽管在理智上往往承认）：网络化系统比只跑在一台机器上的系统**更容易**出故障，而且故障常常是**部分**的而不是整台挂掉。可能一次写入成功、另一次失败，那我们要怎么拿到一致的数据视图？部分失败要难推理得多。

交换机宕机、GC 停顿让 leader「凭空消失」、套接字写入看起来成功其实在另一端失败、某台机器磁盘慢把整个集群里的通信协议都拖慢……凡此种种。读本地内存就是比跨几台交换机更稳。

为失败而设计。

**写出稳健的分布式系统，比写出稳健的单机系统更贵。** 稳健的分布式方案通常要花钱更多：因为只有「很多台机器在一起」时，才会出现某些故障模式。虚拟机和云把成本拉低了一些，但仍比不上在一台你已有的机器上设计、实现和测完那么便宜。也有些故障条件很难在单机环境里复现——要么数据规模大到单机塞不下才会暴露，要么非得要数据中心那种网络条件；分布式系统往往要**真的**铺开运行才能把 bug 清掉，而不是只靠模拟。当然，模拟仍然非常有用。

**稳健的开源分布式系统，远比稳健的单机开源系统少见。** 长期跑一堆机器，对开源社区是负担。爱好者和业余涉猎者是开源的主力军，他们没有财力去摸索、修掉一个分布式系统会遇到的大半问题；大家用业余时间、用自己已有的机器写代码寻开心。愿意起环境、长期维护、为一堆机器付钱的人，在开源里难找得多。

其中一部分缺口由企业的工程师补上了——但他们**组织的优先级**未必与**你组织的优先级**一致。

开源圈里也有人意识到这件事，但还没有称得上解决。这很难。

**协调非常难。** 能不做就尽量别让机器互相协调。这常被说成「水平扩展」。水平扩展真正的窍门是**独立**：把数据送到各台机器上时，让机器之间的通信与共识尽可能少。每多一次「两台机器要对某件事达成一致」，服务就难实现一分。信息传播速度有上限，网络通信比你想象的更不可靠，而你对「共识」的理解也很可能从一开始就不对。这里读一读 [Two Generals](http://en.wikipedia.org/wiki/Two_Generals%27_Problem) 与 [Byzantine Generals](http://en.wikipedia.org/wiki/Byzantine_Generals%27_Problem) 会有帮助。（对了，Paxos [真的很难实现](http://research.google.com/pubs/pub33002.html)；并不是老司机摆架子。）

**问题若能塞进内存，多半算简单题。** 对分布式工程师来说，限制在一台机器上的问题通常都好办。数据要跨越「几台交换机」而不是「几次指针解引用」时，怎么快速处理就难多了。在分布式系统里，从计算机科学发端起就写进教科书的那套「单机效率技巧」往往不再适用。单机上跑的算法，文献和实现都很多——因为大多数计算历史上就是在**单台的、互不协调的机器**上完成的；面向分布式的要少得多。

**「变慢了」是你这辈子最难排的问题之一。** 它可能指完成用户请求所涉及的**多个系统里有一个或多个**慢；也可能指跨多机的流水线里**某个变换环节**慢。它难，一部分是因为这句话几乎不给缺陷位置留线索。那些**部分故障**——在你常看的那几张图上都看不出来——就蹲在暗处。而且在退化变得非常明显之前，你很难拿到足够资源（时间、预算、工具）去解决它。Dapper 和 Zipkin 的出现自有原因。

**在全链路落实反压（backpressure）。** 反压是：服务方如何把「我扛不住了」**信号**传给请求方，以及请求方如何处理这些信号，从而避免把自己和服务方一起压垮。为反压做设计，意味着在过载与故障时段也要**限制资源使用**——这是稳健分布式系统的基本块之一。实现上通常要么在资源吃紧或出错时**丢弃新消息**，要么把错误回给调用方（两种都做**打点**）；对下游连接与请求做**超时**和**指数退避**也必不可少。若没有反压，级联故障或非预期的消息丢失就很容易出现：一个系统扛不住另一个系统的故障时，往往把失败继续传给依赖它的系统。

**尽量做出「部分可用」。** 部分可用 = 系统局部坏了仍能**先返回一部分结果**。

搜索是理想的例子：搜索要在「结果有多好」和「让用户等多久」之间权衡。典型做法是设一个**时间上限**；若搜遍所有文档前超时，就返回**已收集到的**结果。面对间歇性变慢和错误，这就更容易扩展——那些故障被当成「没法搜全文档」的一种；允许向用户返回部分结果，系统的韧性也随之提高。

再想想 Web 应用里的私信功能：无论你多努力，总会出现「存放私信的机器同时挂掉足够多的数量」，用户会察觉。那我们希望这里出现**哪一种**部分故障？

要想一想。人们通常更能接受「私信对自己（或许还有一部分人）暂时不可用」，而不能接受「所有用户都丢了一部分消息」。若系统过载或某台机器挂了，让**一小撮用户**失败，往往优于让**更大比例用户**丢数据。此外，我们可能也不希望**无关功能**（例如公开上传图片）仅仅因为私信出问题就受波及。你愿意花多少力气把这些**故障域**隔开？

能在部分可用里识别这类取舍，是值得放进工具箱的能力。

**度量（metrics）是把活干成的必经之路。** 暴露延迟百分位、特定动作上的递增计数、变化率等指标，是弥合「你以为系统在生产里怎样」与「它实际怎样」之间鸿沟的几乎唯一办法。搞清第 20 天与第 15 天的行为差异，是**像样的工程**与**装神弄鬼**的分野。当然，metrics 是理解与定位问题的必要前提，但光有它并不自动告诉你下一步该做什么。

顺带说一句日志：日志很有用，但也常**骗人**。例如几类错误的日志可能占了文件的很大篇幅，但在请求里实际占比极低。因为大多数情况记成功是浪费（还会把盘打爆），工程师又常猜错哪些错误类值得打，日志就会被各种零碎塞满。更好的做法是：**假设读日志的人没见过你的代码**来记录。

我见过不少故障被人为拖长：另一位工程师（或我自己）在日志里看到点蹊跷就大惊小怪，却**没先和 metrics 对一下**。也见过有人（或我自己）凭寥寥几行日志就**福尔摩斯附体**，推出一整串失败行为。但请注意：a）我们记得这些「成功案例」正是因为它极少；b）若没有 metrics 或实验佐证，你就不是福尔摩斯。

**用百分位，别只用平均值。** 在绝大多数分布式系统里，百分位（P50、P99、P99.9、P99.99）比平均值信息量大得多。用均值隐含假设指标呈**钟形曲线**，可实际上你在乎的指标里很少有谁真是这样。「平均延迟」常被报出来，但我从没见过哪个分布式系统的延迟真呈钟形。若分布不是那样，平均值就没有意义，只会导向错误决策与错误理解。默认谈百分位，你才更贴近用户眼里系统真实的样子。

**学会估算容量。** 你会因此把一天有多少秒背下来。知道完成一项任务需要多少台机器，差别在于：系统是**能长期服役**，还是上岗三个月就得换，或更糟——还没上线生产就得换。

以推文 ID 为例：一台常见机器内存里能塞多少个？2012 年末，一台典型机器大约 24 GB 内存，要给 OS 留 4–5 GB 开销，至少还要几 GB 处理请求，而一个 tweet id 是 8 字节——你会反复做这种信封背面的推算。Jeff Dean 的 [Numbers Everyone Should Know](http://www.cs.cornell.edu/projects/ladis2009/talks/dean-keynote-ladis2009.pdf) 幻灯片很能校准预期。

**特性开关（feature flags）是基础设施怎样滚上线的。** 产品工程师常用特性开关在系统里推出新功能；常见于前端 A/B，只给一部分用户看新界面或新能力。它们同样是**替换底层基础设施**的利器。

太多项目死在「大红按钮式切换」或一连串大红按钮切换上——太晚才发现 bug，只能回滚。用特性开关，你能对项目更有把握，也能把失败成本控制住。

假设你从单一数据库迁到一层对外隐藏新存储细节的服务。可以用开关**慢慢加大**对新服务的写入，与旧库写入**并行**，确认写路径正确且够快。写路径到 100% 且回填完成后，可用另一个开关开始从该服务读，但**先不把数据打进用户响应**，以观察性能；再用一个开关对旧系统与新系统的读结果做比对；最后一个开关逐步放大新系统上的「真实」读流量。

把发布拆成多步、用开关换来**快速、局部的**反应，你就能在爬坡过程中更早发现问题，而不是憋到「大爆炸」那一刻。出事时把开关拧回较低（甚至零）立刻止血；调节速率还能在不同流量档位下调试和实验，且清楚即便出问题也不至于全军覆没。也可以选别的迁移策略（例如按用户切流量）以获得更好可见性。新服务尚在原型阶段时，把开关关小，还能让新系统少吃资源。

当然，对学院派训练出来的人或初心良好的新人而言，特性开关听起来像**可怕的条件分支泥沼**；接受特性开关也意味着承认：**多套基础设施与数据版本并存是常态，而非罕见特例**。这是很深的一课：单机上行得通的手段，遇到分布式问题时往往会踉跄。

最好把特性开关理解为一种取舍：用**局部**（单库、单代码路径）的复杂，换**全局**的简单与韧性。

**审慎选择 ID 空间。** 你为系统选的 ID 空间会塑造系统形态。

要到一份数据所需的 ID **越多**，分区时可选方案就越多；所需 ID **越少**，消费你系统输出就越容易。

以 Twitter API 第一版为例：所有读、写、删推文都围绕**每条推文一个数字 ID**。推文 ID 是单纯的 64 位数，与其他数据不相连。推文一多，就很明显：若同一用户的推文存在同一台机器上，会更容易构建该用户时间线与关注流。

但公开 API 要求**仅凭推文 ID** 就能定位任意推文。若按用户分区推文，就得再做一层**查找服务**，知道某个 tweet id 属于哪个用户——能做，但有实实在在的成本。

另一种 API 设计可以在查找时**强制带 user id**，上线初期只用 tweet id 存，直到按用户分区就绪。再一种做法是把 user id 编进 tweet id 本身，代价是 id 不再 k-sortable、也不再是纯数字。

留神 ID 里**显式或隐式**编码了什么信息。客户端可能利用 ID 结构去匿名化隐私数据、以意外方式爬取你的系统（自增 id 常为痛脚），或实施 [OWASP 所述那一类不安全直接对象引用等攻击](https://www.owasp.org/index.php/Top_10_2010-A4-Insecure_Direct_Object_References)。

**利用数据局部性（data-locality）。** 处理与缓存越靠近数据的持久存储，吞吐往往越高效，一致性缓存也越容易做得又快又稳。比起指针解引用和 `fread(3)`，网络的失败与延迟都多得多。

局部性当然也包括**时间**上的近：多个用户几乎同时发起相同昂贵请求，也许可以合并；同一类数据请求在时间上扎堆，也许可以合成更大的一笔请求——常能换来更低通信开销与更易的故障管理。

**把缓存里的数据写回持久层是糟糕做法。** 这种情况比你想象的多，尤其在最初设计者分布式经验不足的系统里；你接手的系统里会有很多带着这个毛病。若实现者大谈「俄罗斯套娃式缓存」，你撞上显眼 bug 的概率很高。这一条本可省略，但我个人特别痛恨。常见症状是用户信息（昵称、邮箱、哈希后的密码等）**神秘地回到旧值**。

**单台机器能做的比你听说得多。** 一线里有不少关于机器能力的误读，来自经验不多的从业者。

2012 年末，一台轻量 Web 服务器往往有 6 个以上处理器、24 GB 内存，磁盘空间你也用不完。现代运行时上，一台机器跑一个略复杂的 [CRUD](http://en.wikipedia.org/wiki/Create,_read,_update_and_delete) 应用，几百毫秒内**几千 QPS** 只是很宽松的下限。从运维视角看，多数场景下每台机器只有几百 RPS **并不值得吹嘘**。

只要肯 profile、按测量结果加优化，更高性能并不难拿。

**用 CAP 定理来审视系统设计。** CAP 不是用来「搭建」系统的积木，也不是你可以当作第一性原理再推导出一整套可行实现的定理——它概括得太宽，解空间也太大。

但它很适合用来**批评性分析**分布式设计、理解必须做的权衡。把设计拿出来逐项对照 CAP 加在各子系统上的约束，迭代之后通常会更好。课后作业：把 CAP 约束套进真实世界里的「俄罗斯套娃缓存」实现，看会发生什么。

最后一句：在 C、A、P 里，你[没法只要 CA](http://codahale.com/you-cant-sacrifice-partition-tolerance/)。

**抽出独立服务。** 这里的「服务」指「包含比纯存储更高层逻辑、通常是请求–响应式 API 的分布式系统」。留意那些**若代码在单独服务里会更好改**的改动点。

抽服务有类似建库的**封装**收益；但相对发库，抽服务往往能让你**更快、更省事地部署变更**，而不必像升级各客户系统里的依赖库那样协调一堆事情。（当然，若抽出来的服务本身很难部署，那客户系统侧可能反而更好部署。）这是因为小服务里代码与运维依赖更少，边界也更硬，不容易像库那样被「抄近道」——而近道几乎总让 internals 或客户迁到新版更难。

多个客户系统并存时，用服务的**协调成本**也通常低于共享库：即便 API 不变，升级库仍要协调各客户部署顺序；顺序错了还可能**数据损坏**（而且很难事先料到）。若客户由不同维护者负责，升级库的**社交成本**也比部署服务更高：让别人跟上你的节奏很难，因为他们的优先级未必与你一致。

最典型的用例是：**藏住即将大改的存储层**。被抽出的服务 API 比背后存储更易用、表面积更小。客户不必了解向新存储或新格式**缓慢迁移**的复杂度，只要评测新服务里与新布局相伴而生的那些 bug 就行。

这样做牵涉大量运维与人的问题，此处写不尽，得另开一篇。
