## 12 个 MCP 服务器

MCP（Model Context Protocol）是一个开放标准，简化 AI 模型（特别是 LLM）与外部数据源、工具和服务的交互方式。MCP 服务器充当这些 AI 模型和外部工具之间的桥梁。以下是顶级 MCP 服务器：

1. **文件系统 MCP 服务器**
   允许 LLM 直接访问本地文件系统以读取、写入和创建目录。

2. **GitHub MCP 服务器**
   连接 Claude 到 GitHub 仓库，允许文件更新、代码搜索。

3. **Slack MCP 服务器**
   Slack API 的 MCP 服务器，使 Claude 能够与 Slack 工作空间交互。

4. **Google Maps MCP 服务器**
   Google Maps API 的 MCP 服务器。

5. **Docker MCP 服务器**
   与 Docker 集成以管理容器、镜像、卷和网络。

6. **Brave MCP 服务器**
   使用 Brave 的搜索 API 进行网络和本地搜索。

7. **PostgreSQL MCP 服务器**
   启用 LLM 检查数据库模式并执行只读查询的 MCP 服务器。

8. **Google Drive MCP 服务器**
   与 Google Drive 集成的 MCP 服务器，允许读取和搜索文件。

9. **Redis MCP 服务器**
   提供 Redis 数据库访问的 MCP 服务器。

10. **Notion MCP 服务器**
    此项目为 Notion API 实现 MCP 服务器。

11. **Stripe MCP 服务器**
    与 Stripe API 交互的 MCP 服务器。

12. **Perplexity MCP 服务器**
    连接到 Perplexity 的 Sonar API 进行实时搜索的 MCP 服务器。

## MCP vs. A2A 协议

**Model Context Protocol (MCP)** 将 AI 代理连接到外部数据源，如数据库、API 和文件，通过 MCP 服务器，从而用真实世界上下文丰富它们的响应。

**Google 的 Agent-to-Agent (A2A) 协议** 使 AI 代理能够沟通和协作，允许它们委托任务、共享结果，并增强彼此的能力。

### MCP 和 A2A 可以组合成整体架构

以下是它如何工作：

1. **每个 AI 代理（使用工具如 Langchain 与 GPT 或 Claude）通过 MCP 服务器连接到外部工具以进行数据访问**

2. **外部工具可以包括云 API、本地文件、网络搜索或通信平台如 Slack**

3. **同时，AI 代理可以使用 A2A 协议相互通信以协调行动、共享中间输出，并共同解决复杂任务**

此架构实现丰富的外部上下文（通过 MCP）和去中心化代理协作（通过 A2A）。

## 缓存系统如何出错

下图显示缓存可能出错的 4 种典型情况及其解决方案。

### 1. 雷群问题（Thunder herd problem）
当缓存中大量键同时过期时发生这种情况。然后查询请求直接击中数据库，使数据库过载。

**有两种方法缓解此问题**：
- 一是避免为键设置相同的过期时间，在配置中添加随机数
- 二是只允许核心业务数据击中数据库，防止非核心数据在缓存恢复前访问数据库

### 2. 缓存穿透（Cache penetration）
当键在缓存或数据库中都不存在时发生这种情况。应用无法从数据库检索相关数据以更新缓存。此问题对缓存和数据库都造成很大压力。

**解决此问题，有两个建议**：
- 一是为不存在的键缓存空值，避免击中数据库
- 二是使用布隆过滤器首先检查键是否存在，如果键不存在，我们可以避免击中数据库

### 3. 缓存击穿（Cache breakdown）
这类似于雷群问题。当热键过期时发生。大量请求击中数据库。由于热键占用 80% 的查询，我们不为它们设置过期时间。

### 4. 缓存崩溃（Cache crash）
当缓存宕机且所有请求都去数据库时发生这种情况。有两种方法解决此问题：
- 一是设置断路器，当缓存宕机时，应用服务不能访问缓存或数据库
- 二是为缓存设置集群以提高缓存可用性

## 8 个系统设计概念（1 张图）

非功能需求定义系统的质量属性，确保它在现实条件下可靠执行。一些关键 NFR 及实现方法如下：

### 1. 可用性与负载均衡器
可用性确保系统始终保持运行并对用户可访问。使用负载均衡器在多个服务实例间分发流量以消除单点故障。

### 2. 延迟与 CDN
延迟指系统响应用户请求的时间延迟。CDN 通过缓存内容更靠近用户减少延迟。

### 3. 可扩展性与复制
可扩展性是系统通过添加资源处理增加负载的能力。复制跨多个节点分发数据，实现更高吞吐和工作负载。

### 4. 持久性与事务日志
持久性保证一旦数据提交，即使在故障事件中它也保持安全。事务日志持久化所有操作，允许系统在崩溃后重建状态。

### 5. 一致性与最终一致性
一致性意味着所有用户看到相同数据。最终一致性允许临时差异，但随时间同步副本到一致状态。

### 6. 模块化与松耦合高内聚
模块化促进构建具有良好分离和自包含组件的系统。松耦合和高内聚帮助实现相同目标。

### 7. 可配置性
可配置性允许系统轻松调整或修改而无需改变核心逻辑。配置即代码通过版本控制脚本管理基础设施和应用设置。

### 8. 弹性与消息队列
弹性是系统从故障中恢复并继续平稳运行的能力。消息队列解耦组件并缓冲任务，启用重试。

- MCP 简化 AI 与外部工具交互
