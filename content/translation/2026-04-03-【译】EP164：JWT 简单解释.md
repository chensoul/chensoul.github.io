JWT 或 JSON Web Tokens 是开放标准用于安全传输信息在双方之间。它们广泛使用为认证和授权。

## JWT 三个主要组件

每个 JWT 携带三个主要部分：

### 头部（Header）
- 每个 JWT 携带头部指定算法为签署 JWT
- 它以 JSON 格式编写

### 负载（Payload）
- 负载由声明和用户数据组成
- 有不同类型声明如注册、公开和私有声明

### 签名（Signature）
- 签名是使 JWT 安全什么
- 它通过获取编码头部、编码负载、秘密密钥和算法并签署它创建

## JWT 签署方式

JWTs 可以签署在两种不同方式：

### 对称签名（Symmetric Signatures）
- 使用单个秘密密钥为签署令牌和验证它
- 相同密钥必须共享在签署 JWT 服务器和验证它系统之间

### 非对称签名（Asymmetric Signatures）
- 在这种情况下，私钥用于签署令牌，公钥验证它
- 私钥保持安全在服务器上，而公钥可以分发给任何人需要验证令牌

## 5 个 API 设计支柱

API 是现代系统骨干。但也重要设计它们在正确方式。

### 1. 接口（The Interface）
- API 设计关注定义 API 输入和输出
- 例如，定义 CRUD 操作如何暴露给用户或客户端

### 2. API 范式（API Paradigms）
- API 可以构建遵循不同范式，每个有自己协议和标准集
- 一些选项是 REST、GraphQL 和 gRPC

### 3. 关系中 API（Relationships in API）
- API 经常需要建立关系在各种实体之间
- 例如，用户可能有多个订单关联到他们账户
- API 端点应该反映这些关系为更好客户端体验

### 4. 版本控制（Versioning）
- 当修改 API 端点，适当版本控制和支持向后兼容是重要

### 5. 速率限制（Rate Limiting）
- 速率限制用于控制用户可以进行请求数量到 API 在特定时间框架内
- 这对维护 API 可靠性和可用性至关重要

## 计算机内存如何工作

这里简单分解显示数据如何移动通过系统从输入到处理到存储。

### 内存层次
1. **数据进入通过输入源**如键盘、鼠标、相机或远程系统
2. **永久存储**保存你系统文件、应用和媒体。这包括硬盘、USB 驱动器、ROM/BIOS 和基于网络存储
3. **RAM** 是你计算机工作区。它包括物理内存和虚拟内存，临时存储数据和程序当你使用它们
4. **缓存内存**坐更靠近 CPU 并分裂为 Level 1 和 Level 2。它帮助加速访问频繁使用数据
5. **CPU 寄存器**是最快和最小内存单元。它们直接使用由处理器执行指令几乎瞬间

**内存金字塔**：越高你在内存金字塔，越快和越小存储。

## Kubernetes 扩展策略

### 水平 Pod 自动扩展（HPA）
- 水平 Pod 自动器自动扩展 Pods 数量在 Deployment、ReplicaSet 或 StatefulSet
- 基于观察 CPU 利用率、内存使用或自定义指标

### 垂直 Pod 自动扩展（VPA）
- 基于应用要求，VPA 调整资源分配给单独 pods，如 CPU 和内存
- 这个方法动态改变 pod 资源设置基于工作负载指标

### 集群自动扩展
- 集群自动器自动调整节点数量在 Kubernetes 集群
- 它与云提供商交互添加或移除节点基于要求
- 这对维护平衡集群重要

### 预测自动扩展
- 预测自动扩展使用机器学习预测未来资源要求
- 它帮助 Kubernetes 调整资源通过预期工作负载需求

- 输入源 → 永久存储 → RAM → 缓存 → CPU 寄存器

- HPA：水平扩展 Pods
- VPA：垂直扩展资源
- 集群自动：调整节点
- 预测：机器学习预测

- JWT 用于安全认证

- Kubernetes 有多种扩展策略
