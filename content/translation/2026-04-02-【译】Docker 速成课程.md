在软件开发的旧时代，将应用从代码到生产是缓慢和痛苦的。开发者与依赖地狱斗争，因为测试和生产环境在细微方式上不同，导致代码神秘地在一个环境工作但在另一个环境不工作。然后 Docker 在 2013 年出现，最初在 dotCloud 内创建作为容器技术的实验以简化部署。Docker 在那年 3 月开源，在接下来的 15 个月里它崛起为领先的容器平台。

在这份通讯中，我们将探索容器技术的历史、推动 Docker 飞速崛起的具体创新，以及实现其魔力的 Linux 基础。我们将解释 Docker 镜像是什么、它们与虚拟机如何不同，以及你是否需要 Kubernetes 来有效使用 Docker。结束时，你将理解为什么 Docker 已成为云中打包和分发应用的标准。

## 后端基础设施演进

在过去二十年，后端基础设施快速演进：

1. **物理服务器时代**：应用直接在物理服务器（"裸机"）上运行。团队购买、上架、堆叠、通电和配置每台新机器。这非常耗时。

2. **硬件虚拟化**：允许多个虚拟机在单个强大物理服务器上运行。这实现更高效资源利用。但配置和管理 VM 仍需要大量工作。

3. **基础设施即服务（IaaS）**：如 Amazon EC2。IaaS 移除设置物理硬件的需求并提供按需虚拟资源。但开发者仍必须手动配置 VM 与库、依赖等。

4. **平台即服务（PaaS）**：如 Cloud Foundry 和 Heroku 是下一个大转变。PaaS 提供托管开发平台以简化部署。但跨环境不一致导致"在我的机器上工作"问题。

这带我们到 2013 年的 Docker。Docker 通过两个关键创新改进 PaaS。

## Docker vs. 虚拟机

容器技术经常与虚拟机比较，但它们使用非常不同的方法。

### 虚拟机（VM）
- VM 管理程序模拟底层服务器硬件（如 CPU、内存和磁盘），允许多个虚拟机共享相同物理资源
- 它在虚拟化硬件上安装客户操作系统
- 在客户 OS 上运行的进程看不到主机硬件资源或其他 VM

### Docker 容器
- Docker 容器共享主机操作系统内核
- Docker 引擎不虚拟化 OS 资源
- 容器通过 Linux 命名空间和控制组（cgroups）实现隔离
  - **命名空间**：提供进程、网络、挂载和其他资源的隔离
  - **cgroups**：限制和计量容器的 CPU、内存和磁盘 I/O 等资源使用

这使得容器比 VM 更轻量且可移植。多个容器可以共享主机及其资源。它们启动也快得多，因为没有完整 VM OS 启动。

Docker 不是某些人描述的"轻量级虚拟化"。它使用 Linux 原语隔离进程，不像管理程序那样虚拟化硬件。这种 OS 级隔离实现轻量级 Docker 容器。

## Cloud Foundry vs. Docker

在 Docker 2013 年发布之前，Cloud Foundry 是广泛使用的开源 PaaS 平台。许多公司采用 Cloud Foundry 构建自己的 PaaS 产品。

### Cloud Foundry 优势
- 避免供应商锁定 - 在其上构建的应用可跨 PaaS 实现移植
- 支持多样基础设施环境和扩展需求
- 全面支持主要语言（如 Java、Ruby 和 JavaScript）和数据库（如 MySQL 和 PostgreSQL）
- 一套用于部署应用的打包和分发工具

Cloud Foundry 在底层依赖 Linux 容器提供隔离应用沙箱环境。然而，这种支撑 Cloud Foundry 的核心容器技术未作为面向用户的功能暴露或强调为关键架构组件。

提供 Cloud Foundry PaaS 解决方案的公司忽视了将容器解锁为开发者工具的潜力。它们未能认识到容器如何从内部隔离机制转变为外部化打包格式。

### Docker 解决的两个关键 PaaS 打包问题

1. **将应用、配置、依赖项和 OS 捆绑到单个可部署镜像中**
2. **保持本地开发环境与云运行时环境一致**

这优雅地解决了困扰 PaaS 的依赖和兼容性问题。但 Cloud Foundry 没有足够快适应支持 Docker 镜像。这允许 Docker 镜像在云计算环境中激增。

Docker 早期受欢迎是因为它在应用打包和部署方面创新。其初始成功 largely 由于这种在轻量级容器中隔离应用的新颖方法。

## Docker 的演进

随着 Docker 流行度增长，公司寻求将其产品扩展到容器化之外。它冒险扩展到完整 PaaS 平台。这导致开发 Docker Swarm 用于集群管理，以及收购 Fig（后来 Docker Compose）以增强编排能力。

Docker 的抱负引起一些科技巨头注意。Google、RedHat 和其他 PaaS 公司都想加入这个热门新技术。

## 译者总结

这篇文章涵盖了 Docker 和容器技术的核心概念：

**后端基础设施演进**：
| 时代 | 技术 | 特点 |
|------|------|------|
| 物理服务器 | 裸机 | 耗时配置 |
| 硬件虚拟化 | VM | 更高效但重 |
| IaaS | EC2 | 按需资源 |
| PaaS | Cloud Foundry | 简化部署 |
| 容器 | Docker | 轻量、便携 |

**Docker vs. VM**：
| 特性 | VM | Docker |
|------|-----|--------|
| 虚拟化 | 硬件 | OS 级 |
| 隔离 | 管理程序 | 命名空间+cgroups |
| 重量 | 重 | 轻量 |
| 启动 | 慢 | 快 |
| 共享 | 不共享 | 共享主机内核 |

**Docker 关键创新**：
1. 捆绑应用、配置、依赖、OS 到单个镜像
2. 保持本地开发与云环境一致

**关键洞察**：
- Docker 使用 Linux 原语（命名空间、cgroups）隔离
- 不是"轻量级虚拟化"而是 OS 级隔离
- Cloud Foundry 忽视容器作为开发者工具潜力
- Docker 解决 PaaS 打包问题

理解这些概念对理解现代容器化至关重要。
