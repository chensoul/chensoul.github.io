给 LLM 提供更多信息反而会使其变笨。Chroma 2025 年的一项 [研究](https://research.trychroma.com/context-rot) 测试了 18 个最强大的语言模型（包括 GPT-4.1、Claude 和 Gemini），发现所有模型的性能都随着输入量的增加而下降。

这种下降并非微不足道。某些模型在 95% 的准确率下保持稳定，但一旦输入超过某个长度，就会暴跌至 60%。

这一发现打破了关于 LLM 最常见的迷思之一：更多上下文总是更好。现实是，LLM 存在架构盲点，这使得你呈现给它的信息及其结构比包含多少信息重要得多。

这门做对的学科被称为**上下文工程（Context Engineering）**。

在本文中，我们将探讨 LLM 如何处理你给它的信息、什么是上下文工程，以及有助于此的策略。

<!--more-->

在继续之前，有三个术语在谈论 LLM 时经常出现。先搞清楚这些，会让后续内容更容易理解。

- **Tokens（词元）**：LLM 思考的单位。它们不是完整的单词，而是文本块，平均约四分之三个单词。单词 "context" 是一个 token，而 "engineering" 被分成两个。模型处理的每一段文本（从你的问题到其指令，再到任何包含的文档）都以 tokens 计量。
- **Context Window（上下文窗口）**：模型在单次交互中可见的 token 总数。所有内容都必须适应这个窗口：定义模型行为的系统指令、对话历史、注入的外部文档或数据，以及你的实际问题。现代模型的上下文窗口从 128,000 到超过 200 万 tokens 不等。这听起来很大，但正如我们将看到的，更大并不直接意味着更好。
- **Attention（注意力）**：模型用来确定哪些 tokens 对其他 tokens 重要的机制。在生成响应的每个新 token 之前，模型会将其与当前上下文窗口中的每个其他 token 进行比较。这赋予 LLM 在长文本中连接想法的能力，但也是其最重要限制的来源。

当我们向 LLM 发送文本时，它不会像人类那样从上到下阅读。注意力机制将每个 token 与每个其他 token 进行比较以计算关系，这意味着模型原则上可以将输入第一句中的想法与最后一句中的想法连接起来。然而，这种能力伴随着两个关键成本。

- 第一个是计算成本。将上下文窗口中的 token 数量翻倍，计算量大约增加四倍。更长的上下文不成比例地更慢、更昂贵。
- 第二个成本更为关键。注意力并非均匀分布在上下文窗口中。研究一致表明，LLM 对输入**开头和结尾**的 tokens 关注度最高，中间显著下降。这被称为**"迷失在中间"（Lost in the Middle）**问题，研究发现，当相关信息放在输入中间时，准确率比放在开头或结尾下降超过 30%。

这不是任何特定模型的缺陷，而是 transformer（为几乎所有现代 LLM 提供动力的神经网络架构）编码 token 位置的结构性问题。

大多数现代 LLM 使用的**旋转位置编码（RoPE）**方法引入了衰减效应，使远离序列开头和结尾的 tokens 落入低注意力区域。较新的模型减轻了严重程度，但没有生产模型完全消除它。

**实际含义**：输入中信息的位置与信息本身一样重要。如果我们将长文档粘贴到 LLM 中，模型最有可能错过埋在中间页面的信息。

不均匀的注意力分布是一个问题，但还有更广泛的模式会加剧它，称为**上下文腐化（Context Rot）**。

上下文腐化是随着输入长度增加，LLM 性能下降的现象，即使在简单任务上也是如此。Chroma 研究团队 2025 年的研究发现，这种下降不是渐进的。模型可以在达到某个上下文长度之前保持近乎完美的准确率，然后性能不可预测地急剧下降，因模型和任务而异，无法可靠预测何时会达到临界点。

### 为什么会发生？

添加到上下文窗口的每个 token 都从有限的注意力预算中抽取。不相关的信息将重要信息埋在低注意力区域，听起来相关但实际上无用的内容会混淆模型识别相关信息的能力。**模型不会因更多输入而变得更聪明，而是会分心**。

此外，LLM 是**无状态的**。它们在调用之间没有记忆，每次交互都完全重新开始。当我们与 ChatGPT 等多轮对话时，它似乎"记住"了我们之前说的话，那是因为系统每次都将对话历史重新注入上下文窗口。模型本身什么都不记得，这意味着有人或某个系统必须为每次调用决定包含什么信息、省略什么信息，以及如何结构化。

营销与现实之间也存在显著差距。模型宣传百万 token 的上下文窗口，并在这些长度下通过简单基准测试。然而，**有效上下文长度**（模型实际可靠使用信息的长度）通常小得多。通过"大海捞针"测试（在长文档中找到一个植入的句子）与可靠地综合分散在数百页中的信息非常不同。

**上下文工程**是设计、组装和管理 LLM 生成响应前看到的整个信息环境的实践。它超越了编写单个好指令，而是编排填充上下文窗口的所有内容，使模型恰好拥有当前任务所需的信息，不多不少。

要了解这涉及什么，有助于看看什么实际上在上下文窗口内竞争空间。典型 LLM 调用中有六种类型的上下文：

- 系统指令（定义模型行为的规则、角色和指南）
- 用户输入（你的实际问题或命令）
- 对话历史（当前会话的短期记忆）
- 检索知识（从外部来源提取的文档、数据库结果或 API 响应）
- 工具描述（模型可调用的工具定义及使用方法）
- 工具输出（先前工具调用返回的结果）

用户的实际问题通常只占总 token 数的一小部分。

![上下文窗口组成](02.webp)

其余是基础设施，而上下文工程设计的就是这个基础设施。

这也澄清了上下文工程与提示词工程的区别。提示词工程问："我如何措辞指令以获得最佳结果？"另一方面，上下文工程问："模型此刻需要看到什么？我如何动态组装所有信息？"

提示词工程是上下文工程中的一个组件，专注于指令层，而上下文工程 encompass 模型周围的完整信息系统。正如 Andrej Karpathy 在一条 [广泛引用的推文](https://x.com/karpathy/status/1937902205765607626) 中所说，上下文工程是"用恰到好处的信息填充上下文窗口的微妙艺术和科学"。

使用相同模型的两个人可能获得截然不同的结果。模型相同，但上下文不同，而上下文工程是决定性因素。

开发者已收敛出四种管理上下文的 broad 策略，[分类](https://blog.langchain.com/context-engineering-for-agents/) 为：**Write（写入）**、**Select（选择）**、**Compress（压缩）**和**Isolate（隔离）**。每种策略都是对我们已经涵盖的特定约束的直接响应。

**解决的约束**：上下文窗口有限，无状态意味着信息在调用之间丢失。

与其试图将所有内容保留在上下文窗口内，不如将重要信息保存到外部存储，并在需要时带回。这有两种主要形式。

- 第一种是**草稿纸（Scratchpads）**，代理在长时间运行的任务中将中间计划、笔记或推理步骤保存到外部存储。[Anthropic 的多代理研究系统](https://www.anthropic.com/engineering/building-effective-agents) 正是这样做的。首席研究员代理在任务开始时将其计划写入外部存储器，因为如果上下文窗口超过 200,000 tokens，它会被截断，计划将丢失。
- 第二种形式是**长期记忆**，涉及跨会话持久化信息。ChatGPT 从对话中自动生成用户偏好，Cursor 和 Windsurf 学习编码模式和项目上下文，Claude Code 使用 CLAUDE.md 文件作为持久指令记忆。所有这些系统都将外部存储视为真正的记忆层，上下文窗口作为临时工作区。

**解决的约束**：更多上下文不是更好，模型需要正确的信息而非所有可用信息。

这里最重要的技术是**检索增强生成（RAG）**。与其将所有知识塞入上下文窗口，不如将其存储在外部可搜索数据库中。在查询时，仅检索与当前问题最相关的块，并将这些注入上下文，为模型提供有针对性的知识，而无需其他内容的噪音。

![RAG 示意图](03.webp)

选择也适用于工具。当代理有数十个可用工具时，在每个提示中列出每个工具描述会浪费 tokens 并混淆模型。更好的方法是仅检索与当前任务相关的工具描述。

**关键权衡是精度**。如果检索拉入几乎相关但不完全相关的文档，它们会成为干扰项，增加 tokens 并将重要上下文推入低注意力区域。检索步骤本身必须良好，否则整个策略会适得其反。

**解决的约束**：上下文腐化和更多 tokens 的注意力成本升级。

随着代理工作流跨越数十或数百步，上下文窗口会充满累积的对话历史和工具输出。压缩策略在试图保留基本信息的同时减少这种批量。

**对话摘要**是最常见的方法。例如，Claude Code 在上下文达到 95% 容量时触发"自动压缩"过程，将整个交互历史摘要为更短的形式。Cognition（Devin 编码代理背后的公司）[训练了一个单独的专用模型专门用于摘要](https://blog.langchain.com/context-engineering-for-agents/)，在代理到代理边界处。他们专门为这一步构建了一个单独的模型，这告诉我们糟糕的压缩有多严重，因为被摘要掉的具体决定或细节会永久丢失。

更简单的压缩形式包括**修剪**（从历史中删除较旧的消息）和**工具输出压缩**（在详细搜索结果或代码输出进入上下文之前将其简化为要点）。

**解决的约束**：当太多类型的信息在一个窗口中竞争时的注意力稀释和上下文污染。

与其让一个代理在单个膨胀的上下文窗口中处理所有事情，不如将工作分割到多个专用代理，每个代理都有自己干净、专注的上下文。"研究员"代理的上下文加载搜索工具和检索文档，而"作家"代理的上下文加载风格指南和格式规则，这样两者都不会被对方的信息分散注意力。

[Anthropic 用他们的多代理研究系统证明了这一点](https://www.anthropic.com/engineering/building-effective-agents)，其中首席 Opus 4 代理将子任务委托给 Sonnet 4 子代理。尽管使用相同的底层模型系列，但该系统在研究任务上比单个 Opus 4 代理提高了 90.2%。**整个性能增益来自上下文的管理方式，而非更强大的模型**。

这些策略很强大，但涉及没有普遍正确答案的权衡：

- **压缩 vs 信息丢失**：每次摘要时，都有丢失后来证明重要的细节的风险。压缩越激进，节省的 tokens 越多，但永久销毁重要内容的几率越高。
- **单代理 vs 多代理**：Anthropic 的多代理结果令人印象深刻，但其他人（特别是 Cognition）认为，具有良好压缩的单代理提供更稳定的性能和更低的成本。双方都在辩论同一个核心问题：如何有效管理上下文。答案取决于任务复杂度、成本容忍度和可靠性要求。
- **检索精度 vs 噪音**：RAG 添加知识，但不精确的检索会添加干扰项。如果检索的文档不是真正相关的，它们会消耗 tokens 并将重要内容推入低注意力位置，因此检索系统本身必须经过良好工程，否则 RAG 会使情况更糟。
- **成本 vs 丰富度**：每个 token 都花费金钱和处理时间。注意力的不成比例缩放意味着更长的上下文变得昂贵很快，上下文工程部分是一个经济学问题：找出额外 token 的回报何时不再值得成本。

核心结论是：**模型的质量仅与其接收的上下文一样好**。有效使用 LLM 需要思考模型周围的整个系统，而不仅仅是模型本身。

随着模型变得更强大，上下文工程变得更加重要。当模型足够强大时，大多数失败不再是智能失败，而是**上下文失败**——模型本可以正确，但没有得到所需的内容，或者有太多不需要的内容。

策略正在演变，最佳实践随着新模型的发布而修订。然而，有限注意力、位置偏差和无状态的底层约束是架构性的。

## 译者总结

### 一句话核心
LLM 性能会随输入长度增加而下降，上下文工程通过 Write、Select、Compress、Isolate 四大策略管理信息环境，使模型获得恰好所需的信息。

### 要点回顾
- **注意力分布不均**：LLM 最关注输入的开头和结尾，中间信息容易被忽略（"迷失在中间"问题，准确率下降超 30%）
- **上下文腐化**：超过临界点后，模型性能会急剧下降而非渐进降低（从 95% 暴跌至 60%）
- **四大策略**：Write（外部存储）、Select（RAG 检索）、Compress（摘要压缩）、Isolate（多代理隔离）
- **有效上下文长度**远小于宣传的窗口大小，"大海捞针"测试不等于可靠的信息综合能力
- **多代理系统性能提升 90.2%**来自上下文管理方式，而非更强大的模型

### 边界与易误读
- 更多上下文≠更好，这是一个反直觉但关键的发现
- RAG 检索精度至关重要，不精确的检索会适得其反（成为干扰项）
- 压缩越激进，丢失重要细节的风险越高

### 术语与延伸
- **RoPE（旋转位置编码）**：大多数 LLM 使用的位置编码方法，导致注意力衰减效应
- **RAG（检索增强生成）**：仅检索相关文档块注入上下文，而非全部知识
- **上下文工程 vs 提示词工程**：前者问"模型需要看到什么"，后者问"如何措辞指令"
