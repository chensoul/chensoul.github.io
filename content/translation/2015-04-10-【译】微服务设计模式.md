基于微服务的应用应具备哪些特征，在 [Microservices, Monoliths, and NoOps](https://blog.arungupta.me/microservices-monoliths-noops/) 一文中已有归纳：功能分解或领域驱动设计、定义良好的接口、对外显式发布的接口、单一职责原则，以及可能的**多语言（polyglot）**栈。每个服务完全自治且端到端（full-stack）。因此，只要通过定义良好的接口通信，更换某一服务的实现就不会波及其他服务。这类应用优点不少，但绝非 [免费午餐](https://highscalability.com/blog/2014/4/8/microservices-not-a-free-lunch.html)，在 **NoOps** 上要投入大量精力。

不妨假定你已理解（或至少理解其中一部分）搭建这种应用所需的投入，并准备迈出那一步。具体怎么做？架构上如何着手？这些微服务彼此协作时，可遵循哪些**设计模式**？

**应用本身与团队的功能分解**，是落地成功微服务架构的关键。这样你才能获得**松耦合**（例如 REST 接口）与**高内聚**（多个服务可彼此组合，构成更上层的服务或应用）。

从应用里的**动词**（例如 Checkout）或**名词**（例如 Product）入手拆分，是拆解既有系统的有效途径之一。例如「产品」「目录」「结账」可以是三个独立微服务，再彼此配合，交付完整的购物车体验。

功能分解带来敏捷、灵活、可伸缩以及其它种种 “*ilities”，但业务目标仍然是**做出完整应用**。那么，在识别出各微服务之后，又如何把它们**组合**起来实现应用能力？

下文将介绍几种常用的组合模式。

![微服务功能分解与组合关系示意](01.webp)

## 聚合器微服务设计模式

第一个、也可能是最常见的，是**聚合器（Aggregator）**微服务模式。

最简单的形态下，聚合器可以只是一个**网页**：为完成应用所需能力而调用多个服务。由于各服务（服务 A、B、C）都以轻量 REST 暴露，页面能拉取数据并按需处理或展示。若还需要加工——例如对来自各服务的数据套用业务规则——你多半会用一个 **CDI Bean**（Contexts and Dependency Injection）把数据变换成页面可读的形式。

![微服务聚合器设计模式](02.webp)

聚合器还有另一种用法：**不需要界面**。此时它是一个**更高层的复合微服务**，供其它服务消费。聚合器从各底层微服务收集数据、施加业务规则，再**发布为 REST 端点**，由需要它的服务来调用。

该模式体现 **DRY**（不要重复自己）思想：若有多处都要访问服务 A、B、C，较稳妥的做法是把这部分逻辑抽象进**一个复合微服务**，集中聚合。这样抽象带来的好处是：服务 A、B、C 仍可**独立演进**，业务需求仍由复合微服务统一满足。

注意：每个微服务通常自有（可选的）缓存与数据库。若聚合器本身就是复合微服务，它也可以有自己的缓存与数据库层。

聚合器还可在 **X 轴**（水平复制）与 **Z 轴**（按数据分片等）上**独立扩展**。若聚合器是网页，可多启 Web 服务器；若是基于 Java EE 的复合微服务，也可多启 **WildFly** 实例以应对增长。

## 代理微服务设计模式

**代理（Proxy）**模式可视为聚合器的变体：客户端侧**不必做聚合**，而是按业务需要**改调不同的微服务**。

![微服务代理设计模式](03.webp)

与聚合器类似，代理同样能在 X 轴与 Z 轴上独立伸缩。适合用于：**不希望每个底层服务直接暴露给消费者**，而由一层统一接口对外。

代理可以是**哑代理（dumb proxy）**：把请求**原样转发**给其中一个服务。也可以是**智能代理（smart proxy）**：在响应返回客户端之前做**数据变换**。典型例子是用智能代理封装**面向不同终端的展示层**。

## 链式微服务设计模式

**链式（Chained）**模式会对请求给出**单一的汇总响应**。客户端请求先由服务 A 处理，A 再调用 B，B 又可能调用 C。链上服务通常采用**同步**的 HTTP 请求/响应。

![微服务链式设计模式](04.webp)

务必记住：在整条请求/响应链完成之前——即服务 A ↔ B、B ↔ C 的往返都结束之前——**客户端一直处于阻塞**。B 调 C 的请求，**完全可以**与 A 调 B 的请求长得不一样；B 对 A 的响应，也可以与 C 对 B 的响应大相径庭——这正是各服务**分别叠加自身业务价值**的意义所在。

另一条要点是：**不要把链拉得太长**。同步链路一长，客户端（尤其是等待页面渲染的浏览器）会觉得**久等**。针对这种阻塞式请求/响应，后续模式会谈到变通做法。

仅含**一个**微服务的链称为**单例链（singleton chain）**——日后仍可能向链上继续扩展。

## 分支微服务设计模式

**分支（Branch）**模式在聚合器思路之上，允许**同时处理**两条（往往**互斥**的）微服务链的响应；也可按业务需要，选择调用不同链或**单条**链。

![微服务分支设计模式](05.webp)

服务 A（网页或复合微服务）可以**并行**触发两条链——此时整体很像聚合器；也可以依据客户端请求**只走其中一条链**。

可通过 **JAX-RS** 或 **Apache Camel** 端点的路由来配置，并应支持**动态**调整。

## 共享数据微服务设计模式

微服务的设计原则之一是**自治**：服务端到端自控，掌握 UI、中间件、持久化、事务等组件，因而可以多语言、**因事择器**。例如，若数据更适合放进 **NoSQL**，就不必硬塞进 SQL。

但从既有**单体**演进时，常见痛点是数据库规范化：每个微服务只应拥有**恰如其分**的数据——不多不少。即便单体里只用 SQL，**非规范化**也会带来冗余与潜在不一致。在过渡阶段，有些应用可采用**共享数据**微服务模式。

在该模式中，链路上的若干微服务可能**共享缓存与数据库**。这仅在双方**强耦合**时才有意义。有人视其为反模式，但业务在特定阶段也许不得不如此；对**全新建设、一开始就按微服务建模**的应用，它**确实**是反模式。

![微服务分支共享数据设计模式](06.webp)

它也可视为驶向**完全自治**之前的**过渡形态**。

## 异步消息微服务设计模式

REST 非常普及、也易于理解，但默认是**同步、阻塞**的。异步不是做不到，却往往要**按应用自己约定**去做，因此有的架构会改用**消息队列**，替代纯 REST 的请求/响应。

![微服务异步消息设计模式](07.webp)

在该模式中，服务 A **同步**调用服务 C，C 再通过**共享消息队列**与 B、D **异步**通信。A → C 这一段也可以改成异步（例如用 **WebSocket**）以满足伸缩目标。REST 请求/响应与 **发布/订阅** 混搭，也常能凑齐业务所需。

想了解更多消息形态取舍，可读：[Coupling vs Autonomy in Microservices](https://www.voxxed.com/blog/2015/04/coupling-versus-autonomy-in-microservices/)。
