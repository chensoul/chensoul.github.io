Redis（Remote Dictionary Server）是一个开源（BSD 许可）的内存数据库，常用作缓存、消息代理或流式引擎。它拥有丰富的数据结构支持，包括基本数据结构如 String、List、Set、Hash、SortedSet，以及概率数据结构如 Bloom Filter 和 HyperLogLog。

Redis 非常快。我们可以用其自带工具运行 [Redis 基准测试](https://redis.io/docs/management/optimization/benchmarks/)。吞吐量可以达到每秒近 10 万次请求。

在本期中，我们将讨论为什么 Redis 在其架构设计中如此快速。

## Redis 架构

Redis 是内存键值存储。有几个重要功能：

- 值使用的数据结构
- 数据结构允许的操作
- 数据持久化
- 高可用性

下面是 Redis 架构的高级图。让我们逐一了解。

### 客户端

有两种类型的客户端访问 Redis：一种支持与 Redis 数据库的连接，另一种在前者之上构建并支持对象映射。

Redis 支持广泛的语言，允许它在各种应用中使用。此外，OM 客户端库允许我们建模、索引和查询文档。

### 数据结构

Redis 对值数据类型有丰富支持，包括 Strings、Lists、Sets、Hashes 等。因此，Redis 适合广泛的业务场景。根据数据类型，Redis 支持不同操作。

基本操作类似于关系数据库，支持 CRUD（创建 - 读取 - 更新 - 删除）：

- **GET**：检索键的值
- **PUT**：创建新键值对或更新现有键
- **DELETE**：删除键值对

数据结构和操作是 Redis 如此高效的重要原因。

### 内存存储

Redis 将数据保存在内存中。内存中的数据读写通常比磁盘读写快 1,000 倍 - 10,000 倍。

然而，如果服务器宕机，所有数据都会丢失。所以 Redis 也设计了磁盘持久化以快速恢复。

### 持久化选项

Redis 有 4 种持久化选项：

1. **AOF（Append Only File）**
   AOF 像提交日志一样工作，记录每个写入 Redis 的操作。所以当服务器重启时，写入操作可以重放并重建数据集。

2. **RDB（Redis Database）**
   RDB 在预定义间隔执行时间点快照。

3. **AOF 和 RDB 结合**
   这种持久化方法结合 AOF 和 RDB 的优点。

4. **无持久化**
   持久化可以在 Redis 中完全禁用。这有时用于 Redis 作为较小数据集缓存时。

### 高可用性

Redis 使用主从复制实现高可用性。我们可以配置多个副本用于读取以处理并发读取请求。这些副本在重启后自动连接到主节点并持有领导者实例的精确副本。

当不使用 Redis 集群时，Redis Sentinel 提供高可用性，包括故障转移、监控和配置管理。

### 安全性

Redis 常用作缓存并可保存敏感数据，所以设计为通过受信任客户端在受信任环境中访问。Redis 安全模块负责管理访问控制层并授权对数据执行的有效操作。

Redis 还提供用于配置和管理集群的管理界面。持久化、复制和安全配置都可以通过管理界面完成。

## 为什么 Redis 快速

Redis 不是市场上唯一的内存数据库产品。但它如何实现微秒级数据访问延迟并成为许多公司的热门选择？

一个重要原因是，将数据存储在内存中允许更灵活的数据结构。这些数据结构不需要像普通磁盘数据结构那样经历序列化和反序列化过程，因此可以针对快速读写优化。

### 哈希表

Redis 使用哈希表保存所有键值对。哈希表中的元素保存指向键值对条目的指针。

使用哈希表，我们可以用 **O(1)** 时间复杂度查找键值对。

像所有哈希表一样，当键数量不断增长时，可能出现哈希冲突，意味着不同键落入同一个哈希桶。Redis 通过链接同一哈希桶中的元素解决此问题。当链变得太长时，Redis 将通过利用两个全局哈希表执行重新哈希。

### 数据结构实现

String 类型只有一种实现：**SDS（Simple Dynamic Strings）**。List、Hash、Set 和 SortedSet 都有两种类型的实现。

注意 Redis 7.0 将 List 实现改为 quicklist，ZipList 被 listpack 取代。

### SDS（Simple Dynamic Strings）

Redis SDS 存储字节序列。它以二进制方式操作存储在 buf 数组中的数据，所以 SDS 不仅可以存储文本，还可以存储音频、视频和图像等二进制数据。

SDS 上的字符串长度操作具有 **O(1)** 时间复杂度，因为长度记录在 len 属性中。SDS 预分配空间，free 属性记录未来使用的可用空间。因此 SDS API 是安全的，没有溢出风险。

### Zip List

Zip list 类似于数组。数组的每个元素保存一条数据。然而，与数组不同，zip list 在头部有 3 个字段：

- **zlbytes** - 列表的长度
- **zltail** - 列表末尾的偏移量
- **zllen** - 列表中的条目数

Zip list 在末尾还有一个 zlend，表示列表的结束。

在 zip list 中，定位第一个或最后一个元素是 **O(1)** 时间复杂度，因为我们可以通过头部字段直接找到它们。定位其他元素需要逐个遍历元素，时间复杂度是 **O(N)**。

## 译者总结

这篇文章涵盖了 Redis 的核心概念：

**Redis 定义**：
- 开源内存数据库（BSD 许可）
- 常用作缓存、消息代理、流式引擎
- 吞吐量可达每秒 10 万次请求

**核心功能**：
| 功能 | 说明 |
|------|------|
| 数据结构 | String、List、Set、Hash、SortedSet |
| 操作 | CRUD（GET、PUT、DELETE） |
| 持久化 | AOF、RDB、两者结合、无持久化 |
| 高可用 | 主从复制、Sentinel |

**持久化选项**：
| 选项 | 说明 |
|------|------|
| AOF | 记录每个写入操作，像提交日志 |
| RDB | 预定义间隔快照 |
| AOF+RDB | 结合两者优点 |
| 无持久化 | 适合缓存小数据集 |

**为什么 Redis 快速**：
1. **内存存储**：比磁盘快 1,000-10,000 倍
2. **哈希表**：O(1) 查找时间复杂度
3. **灵活数据结构**：无需序列化/反序列化
4. **SDS**：预分配空间、O(1) 长度操作
5. **Zip List**：O(1) 首尾元素定位

**关键洞察**：
- Redis 速度来自内存存储和高效数据结构
- 哈希表实现 O(1) 查找
- SDS 安全且无溢出风险
- 持久化可选，根据场景选择

理解这些概念对有效使用 Redis 至关重要。
