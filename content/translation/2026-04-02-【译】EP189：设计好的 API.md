## 如何设计好的 API

设计良好的 API 感觉不可见，它只是工作。在那简单性背后是一套一致的设计原则，使 API 可预测、安全和可扩展。

### 以下是好 API 与糟糕 API 的区别

#### 1. 幂等性（Idempotency）
GET、HEAD、PUT 和 DELETE 应该是幂等的。发送相同请求两次，获得相同结果。没有意外副作用。POST 和 PATCH 不是幂等的。每个调用创建新资源或不同地修改状态。

**使用存储在 Redis 或数据库中的幂等键**。客户端用相同键重试，服务器识别它并返回原始响应而不是再次处理。

#### 2. 版本控制（Versioning）

#### 3. 基于名词的资源名称（Noun-based resource names）
资源应该是名词，不是动词。"/api/products"，不是"/api/getProducts"。

#### 4. 安全（Security）
用适当认证保护每个端点。Bearer 令牌（如 JWT）包括头部、负载和签名以验证请求。始终使用 HTTPS 并在每个调用上验证令牌。

#### 5. 分页（Pagination）
返回大数据集时，使用分页参数如"?limit=10&offset=20"以保持响应高效和一致。

## 大数据管道速查表

每个平台提供覆盖整个生命周期的全面服务套件：

- **摄入（Ingestion）**：从各种来源收集数据
- **数据湖（Data Lake）**：存储原始数据
- **计算（Computation）**：处理和分析数据
- **数据仓库（Data Warehouse）**：存储结构化数据
- **展示（Presentation）**：可视化和报告洞察

### AWS
使用 Kinesis 进行数据流，S3 进行存储，EMR 进行处理，RedShift 进行仓库，QuickSight 进行可视化。

### Azure
管道包括 Event Hubs 进行摄入，Data Lake Store 进行存储，Databricks 进行处理，Cosmos DB 进行仓库，Power BI 进行展示。

### GCP
提供 PubSub 进行数据流，Cloud Storage 进行数据湖，DataProc 和 DataFlow 进行处理，BigQuery 进行仓库，Data Studio 进行可视化。

## 如何学习 AWS

AWS 是最流行的云平台之一。当 AWS 宕机时，互联网大部分宕机。

以下是帮助你掌握 AWS 的学习路线图：

### 1. AWS 基础
包括"什么是 AWS？"、全球基础设施、AWS 计费、管理和 IAM 基础等主题。

### 2. 核心计算、存储和网络
包括计算服务如 EC2、Lambda、ECS、EKS，存储服务（如 S3、EBS、EFS、Glacier），和网络服务（如 VPC、ELB、Route 53）。

### 3. 数据库和数据服务
包括关系数据库（RDS MySQL 和 PostgreSQL）、NoSQL，和内存数据库如 ElastiCache（Redis 和 Memcached）等主题。

### 4. 安全、身份和合规
包括 IAM 深入、加密（KMS、S3 SSE）、安全工具、VPC 安全组，和 HIPAA、SOC、GDPR 相关合规工具等主题。

### 5. DevOps、监控和自动化
包括 DevOps 工具（CodeCommit、CodeBuild、CodePipeline）、基础设施即代码、CI/CD 管道、监控工具（CloudWatch、CloudTrail），和成本管理和计费仪表板等主题。

### 6. 学习路径和认证
包括 AWS 学习资源，如 Skill Builder 和文档，和认证路径，如云从业者、解决方案架构师助理、开发者助理、SysOps 和 DevOps 工程师。

## AI 代理技术栈

### 1. 基础模型（Foundation Models）
大规模预训练语言模型，作为 AI 代理的"大脑"，使推理、文本生成、编码和问答等能力成为可能。

### 2. 数据存储（Data Storage）
此层处理向量数据库和内存存储系统，AI 代理用于存储和检索上下文、嵌入或文档。

### 3. 代理开发框架（Agent Development Frameworks）
这些框架帮助开发者构建、编排和管理多步 AI 代理及其工作流。

### 4. 可观察性（Observability）
此类别实现实时监控、调试和记录 AI 代理行为和性能。

### 5. 工具执行（Tool Execution）
这些平台允许 AI 代理与现实世界工具（例如，API、浏览器、外部系统）接口以完成复杂任务。

### 6. 记忆管理（Memory Management）
这些系统管理代理的长期和短期记忆，帮助它们保留有用上下文并从过去交互中学习。

## 如何在 AWS 上构建基本 RAG 应用

RAG 是一种 AI 模式，结合搜索步骤与文本生成。它从知识源（如向量数据库）检索相关信息，然后使用 LLM 生成准确、上下文感知的响应。

### 摄入阶段

1. **所有原始文档（PDF、文本等）首先存储在 Amazon S3 中**
2. **当文件添加时，AWS Lambda 运行摄入函数**
   此函数清理并将文档分割成更小块。
3. **每个块发送到 Amazon Bedrock 的 Titan 嵌入模型**
   将其转换为向量表示。
4. **这些嵌入与元数据一起存储在向量数据库中**
   如 OpenSearch serverless、DynamoDB。

### 查询阶段

1. **用户通过应用前端发送问题**
   到 API Gateway 然后 Lambda 查询函数。
2. **问题使用 Amazon Bedrock Titan Embeddings 转换为嵌入**
3. **此嵌入与向量数据库中存储的文档嵌入比较**
   找到最相关块。
4. **相关块和用户问题发送到 LLM**
   （如 Bedrock 上的 Claude 或 OpenAI）生成答案。
5. **生成的响应通过相同 API 发送回用户**

## 虚拟化类型

虚拟化不仅使服务器高效，它改变了我们构建、扩展和部署一切的方式。以下是现代系统中四种主要虚拟化类型的快速分解：

### 1. 传统（裸机）
应用直接在操作系统上运行。无虚拟化层，进程间无隔离。所有应用共享相同 OS 内核、库和资源。

### 2. 虚拟化（基于 VM）
每个 VM 运行自己完整的操作系统。管理程序坐在物理硬件上并为每个客户机 OS 模拟整机。每个 VM 认为它有专用硬件，即使它共享相同物理服务器。

### 3. 容器化
容器共享主机操作系统的内核但获得隔离运行时环境。每个容器有自己的文件系统，但它们都使用相同底层 OS。容器引擎（Docker、containerd、Podman）管理生命周期、网络和隔离，无需为每个应用需要单独操作系统。轻量且快速。容器在毫秒内启动，因为你不必启动 OS。资源使用比 VM 显著更低。

### 4. 容器在 VM 上
这实际在生产云环境中运行。容器在 VM 内，从两者获得好处。每个 VM 运行自己带有容器引擎的客户机 OS。管理程序提供 VM 间硬件级隔离。容器引擎在 VM 内提供轻量应用隔离。

这是 AWS、Azure 和 GCP 上 Kubernetes 集群背后的架构。你的 pod 是容器，但它们在 VM 内运行，你从未直接看到或管理。

## 译者总结

这篇文章涵盖了 API 设计、大数据管道、AWS 学习、AI 代理、RAG 和虚拟化的核心概念：

**好 API 的 5 大原则**：
| 原则 | 说明 |
|------|------|
| 幂等性 | GET/PUT/DELETE 应幂等 |
| 版本控制 | 支持向后兼容 |
| 名词资源名 | /api/products 不是/api/getProducts |
| 安全 | JWT、HTTPS、令牌验证 |
| 分页 | limit/offset 参数 |

**大数据管道 5 阶段**：
| 阶段 | AWS | Azure | GCP |
|------|-----|-------|-----|
| 摄入 | Kinesis | Event Hubs | PubSub |
| 数据湖 | S3 | Data Lake Store | Cloud Storage |
| 计算 | EMR | Databricks | DataProc/DataFlow |
| 仓库 | RedShift | Cosmos DB | BigQuery |
| 展示 | QuickSight | Power BI | Data Studio |

**AWS 学习 6 大领域**：
1. AWS 基础
2. 核心计算、存储、网络
3. 数据库和数据服务
4. 安全、身份、合规
5. DevOps、监控、自动化
6. 学习路径和认证

**AI 代理技术栈 6 层**：
| 层 | 用途 |
|------|------|
| 基础模型 | AI"大脑" |
| 数据存储 | 向量数据库、记忆 |
| 开发框架 | 构建编排代理 |
| 可观察性 | 监控调试 |
| 工具执行 | 接口外部工具 |
| 记忆管理 | 长短期记忆 |

**RAG 应用 AWS 构建 2 阶段**：
| 阶段 | 步骤 |
|------|------|
| 摄入 | S3 → Lambda → Bedrock → 向量 DB |
| 查询 | 前端→API→Lambda→Bedrock→向量 DB→LLM→响应 |

**虚拟化 4 种类型**：
| 类型 | 特点 |
|------|------|
| 裸机 | 无虚拟化、共享 OS |
| VM | 完整 OS、管理程序隔离 |
| 容器 | 共享内核、轻量快速 |
| 容器在 VM 上 | 生产云架构、Kubernetes |

**关键洞察**：
- API 设计需要一致性和安全性
- 大数据管道跨云类似
- AWS 学习需要系统化路径
- RAG 结合检索和生成
- 容器在 VM 上是生产标准

理解这些概念对构建现代系统至关重要。
