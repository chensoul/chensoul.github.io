## 20 个系统设计概念

以下是 20 个你应该知道的系统设计概念：

1. **负载均衡（Load Balancing）**
   在多个服务器之间分发流量以提高可靠性和可用性。

2. **缓存（Caching）**
   将频繁访问的数据存储在内存中以实现更快访问。

3. **数据库分片（Database Sharding）**
   拆分数据库以处理大规模数据增长。

4. **复制（Replication）**
   跨副本复制数据以实现可用性和容错。

5. **CAP 定理（CAP Theorem）**
   一致性、可用性和分区容错之间的权衡。

6. **一致性哈希（Consistent Hashing）**
   在动态服务器环境中均匀分发负载。

7. **消息队列（Message Queues）**
   使用异步事件驱动架构解耦服务。

8. **速率限制（Rate Limiting）**
   控制请求频率以防止系统过载。

9. **API 网关（API Gateway）**
   路由 API 请求的集中入口点。

10. **微服务（Microservices）**
    将系统分解为独立、松散耦合的服务。

11. **服务发现（Service Discovery）**
    在分布式系统中动态定位服务。

12. **CDN**
    从边缘服务器交付内容以提高速度。

13. **数据库索引（Database Indexing）**
    通过索引重要字段加速查询。

14. **数据分区（Data Partitioning）**
    跨节点划分数据以实现可扩展性和性能。

15. **最终一致性（Eventual Consistency）**
    保证分布式数据库中随时间一致性。

16. **WebSockets**
    启用双向通信以实现实时更新。

17. **可扩展性（Scalability）**
    通过升级或添加机器增加容量。

18. **容错（Fault Tolerance）**
    确保硬件/软件故障期间的系统可用性。

19. **监控（Monitoring）**
    跟踪指标和日志以了解系统健康。

20. **认证和授权（Authentication & Authorization）**
    控制用户访问并安全验证身份。

## 什么是 AI 代理？

AI 代理是一个软件程序，可以与环境交互、收集数据，并使用这些数据实现预定目标。AI 代理可以选择执行最佳行动以实现这些目标。

### AI 代理的关键特征

- **代理可以执行自主行动**
  无需持续人类干预。此外，它们可以有人类在循环中保持控制。

- **代理有记忆**
  存储个人偏好并允许个性化。它也可以存储知识。LLM 可以进行信息处理和决策功能。

- **代理必须能够感知和处理环境中可用的信息**

- **代理也可以使用工具**
  如访问互联网、使用代码解释器和进行 API 调用。

- **代理也可以与其他代理或人类协作**

### 多种类型的 AI 代理可用

如学习代理、简单反射代理、基于模型的反射代理、基于目标的代理和基于效用的代理。

### 具有 AI 代理的系统可以用不同的架构方法构建

- **单代理**
  代理可以作为个人助理服务。
- **多代理**
  代理可以以协作或竞争方式相互交互。
- **人机**
  代理可以与人类交互以更有效地执行任务。

## 基本 Git 速查表

### 基本命令
- `git init` - 初始化新 Git 仓库
- `git clone <repo_url>` - 克隆远程仓库
- `git status` - 检查工作目录状态
- `git add <file>` - 暂存更改以提交
- `git commit -m "message"` - 提交暂存更改
- `git push` - 推送本地提交到远程仓库
- `git pull` - 获取并合并远程仓库更改
- `git diff` - 显示工作目录中的更改
- `git diff --staged` - 显示暂存区和上次提交之间的更改

### 分支和合并
- `git branch` - 列出分支
- `git branch <branch_name>` - 创建新分支
- `git checkout <branch_name>` - 切换到另一个分支
- `git checkout -b <branch_name>` - 创建并切换到新分支
- `git merge <branch_name>` - 合并分支到当前分支
- `git branch -d <branch_name>` - 合并后删除分支
- `git branch -D <branch_name>` - 强制删除分支

### 同步
- `git fetch` - 从远程下载更改不合并
- `git rebase <branch>` - 在另一个分支上重新应用提交
- `git pull --rebase` - 获取并重新应用更改
- `git remote add <name> <url>` - 添加新远程仓库

### 高级 Git
- `git stash` - 临时保存更改不提交
- `git stash pop` - 重新应用暂存更改
- `git cherry-pick <commit>` - 应用特定提交到当前分支
- `git log --oneline` - 查看简化提交历史
- `git reflog` - 显示引用更改历史
- `git log --graph --decorate --all` - 显示可视化提交历史

### 撤销更改
- `git reset <file>` - 取消暂存文件
- `git reset --soft <commit>` - 重置到提交但保留更改
- `git reset --hard <commit>` - 完全重置到上次提交
- `git revert <commit>` - 创建新提交撤销特定提交

### 与他人协作
- `git fork` - 在 GitHub 上 fork 仓库
- `git pull origin <branch>` - 从原始远程分支拉取更改
- `git push origin <branch>` - 推送分支到原始仓库

## 9 个 OOP 设计模式

这些模式可以分为 3 类：

### A - 创建型模式
处理对象创建机制，将客户端代码与具体类解耦。

1. **工厂模式（Factory Pattern）**
   集中对象创建逻辑，基于输入返回不同子类

2. **单例模式（Singleton Pattern）**
   确保类只有一个实例，并提供全局访问

3. **构建器模式（Builder Pattern）**
   逐步构建复杂对象，允许可选配置

### B - 结构型模式
帮助将类和对象组合成更大结构。

4. **适配器模式（Adapter Pattern）**
   允许不兼容接口一起工作

5. **装饰器模式（Decorator Pattern）**
   动态为对象添加新行为

6. **代理模式（Proxy Pattern）**
   充当访问另一个对象的占位符

### C - 行为型模式
专注于对象之间的通信和交互。

7. **策略模式（Strategy Pattern）**
   允许在运行时从可互换策略家族中选择算法

8. **观察者模式（Observer Pattern）**
   启用一对多依赖，当一个对象变化时通知所有依赖

9. **命令模式（Command Pattern）**
   对象封装执行动作或触发事件所需的所有信息

## 面向对象编程的四大支柱

抽象、封装、继承和多态是面向对象编程的四大支柱。它们意味着什么？

### 1. 抽象（Abstraction）
这是隐藏实现细节并仅显示对象基本特征的过程。例如，具有抽象停止方法的 Vehicle 类。

### 2. 封装（Encapsulation）
它涉及将数据（字段）和方法包装在单个单元（类）中，并使用访问修饰符限制直接访问。例如，带有公共 getter 和 setter 的私有字段。

### 3. 继承（Inheritance）
从现有类（父类）创建新类（子类）的过程，继承属性和方法，从而促进代码重用。例如，Car 类继承自 Vehicle 类。

### 4. 多态（Polymorphism）
它允许方法根据它们被调用的对象表现不同。当两个类型共享继承链时，它们可以互换使用而不出错。
