为什么相同的代码在 GPU 上运行快，在 CPU 上慢，在 TPU 上两者都领先？答案是架构。CPU、GPU 和 TPU 设计用于不同的工作负载。

## CPU（中央处理器）

CPU 处理通用计算。它构建用于低延迟和复杂控制流、分支逻辑、系统调用、中断和决策重的代码。

操作系统、数据库和大多数应用在 CPU 上运行，因为它们需要这种灵活性。

## GPU（图形处理器）

GPU 工作方式不同。它不是几个核心，而是将工作分布在数千个核心上，这些核心在超大数据集上执行相同的指令（SIMT/SIMD 风格）。

如果你的工作负载是重复性的，如矩阵数学、像素着色、张量操作，GPU 可以快速处理它。

## TPU（张量处理器）

TPU 是专用硬件。架构围绕使用脉动阵列的矩阵乘法设计，具有编译器控制的数据流和用于权重和激活的片上缓冲区。

它们在神经网络训练和推理上很快，只要工作负载适合硬件。

## GPU 如何工作

当人们说 GPU 强大时，他们真正的意思是：GPU 从头开始构建用于大规模并行。

让我们分解底层发生的事情。

### 顶层结构

在顶层，GPU 芯片由许多流式多处理器（SM）组成。将 SM 视为在芯片上复制的迷你并行引擎。你不是一个大脑，而是数十个更小的同时工作。

### 在每个 SM 内部

- **Warp 调度器**：决定哪组线程（warp）接下来运行
- **数十个 CUDA 核心**：并行执行指令
- **寄存器文件**：以超低延迟存储线程本地数据
- **加载/存储单元**：在寄存器和内存之间移动数据
- **纹理单元**：处理专用内存操作
- **L1 缓存**：提供快速的片上 SM 数据访问

每个 SM 独立工作，但它们通过片上互连连接。在它下面是 L2 缓存，在所有 SM 之间共享。这是协调层。如果一个 SM 在 L1 未命中，它在去全局内存之前检查 L2。

然后是内存控制器，与全局内存接口。这是有趣的地方：

- **极高带宽**
- **比片上内存高得多的延迟**

这就是为什么 GPU 依赖大规模并行。当一些线程等待内存时，数千个其他线程继续执行。

## OAuth 2 工作流

### 授权码流程（+ PKCE）- 用于用户登录

1. 用户请求受保护资源
2. 服务器重定向到授权服务器（IdP）
3. 客户端生成 code_verifier 和 code_challenge（PKCE）
4. 用户认证并给予同意
5. IdP 返回授权码
6. 服务器交换代码（带有验证器）获取令牌
7. 服务器验证令牌并创建会话

PKCE 防止拦截的授权码被重用。这就是为什么它是现代 Web 和移动应用的默认设置。

### 客户端凭据流程 - 用于服务到服务

1. 服务请求访问令牌
2. IdP 认证客户端
3. 它颁发令牌
4. 服务使用 Bearer 令牌调用 API

没有用户。只是机器身份。

## 分布式追踪如何工作

1. **服务生成遥测数据**（追踪、日志、指标）当它们处理请求时
2. **OpenTelemetry 收集器**从所有服务以统一格式接收此数据
3. **收集器将数据分成三个流**：追踪、日志和指标
4. **每个流发送到接收和处理单元**，为存储和分析准备
5. **处理的数据存储在日志数据库**中用于查询和长期访问
6. **来自数据库的数据通过可视化仪表板可视化**用于监控和调试

## API 网关的 4 个用例

API 网关位于你的客户端和服务之间，它做的远不止路由。以下是它真正重要的四个用例：

### 1. 处理流量峰值

没有限速率，一个行为不端的客户端可以搞垮你的整个系统。API 网关基于你定义的策略执行速率限制，如每用户、每 IP、每订阅层级等。超出限制？你得到 429 Too Many Requests。你的后端服务甚至看不到那个流量。它们保持健康而网关承受打击。

### 2. 保护公共 API

每个请求首先到达网关。它检查 bearer 令牌，针对身份提供商（IdP）验证它，并在那里做出访问决定。不需要在每个微服务中复制认证逻辑。网关在一个地方处理 AuthN 和 AuthZ。

### 3. 减少客户端 - 服务器往返

仪表板页面可能需要用户数据、订单历史和支付信息。没有网关，客户端进行三次单独调用。使用请求聚合，客户端发送一个 GET /dashboard 请求。网关在内部扇出到 user-service、order-service 和 payment-service，然后将所有内容组合成单个 JSON 响应返回给客户端。

### 4. 支持多个客户端

Web 和移动应用不需要相同的数据。Web 客户端可能调用 GET /v2/home 并获得带有完整详细信息的丰富负载。移动客户端命中 GET /v1/home 并获得不耗尽数据的较轻响应。网关处理版本控制和负载转换，因此你的后端服务不需要知道哪个客户端在调用。

## 译者总结

这篇文章涵盖了硬件架构和 API 网关的核心概念：

**处理器对比**：
| 类型 | 设计目标 | 适用场景 |
|------|----------|----------|
| CPU | 低延迟、复杂控制流 | 操作系统、数据库、通用应用 |
| GPU | 大规模并行 | 矩阵数学、图形、张量操作 |
| TPU | 专用神经网络 | 神经网络训练和推理 |

**GPU 架构**：
- 流式多处理器（SM）
- Warp 调度器、CUDA 核心
- 寄存器文件、L1/L2 缓存
- 高带宽、高延迟的全局内存

**OAuth 2 流程**：
- 授权码流程（+ PKCE）用于用户登录
- 客户端凭据流程用于服务到服务

**分布式追踪**：
- 服务生成遥测数据
- OpenTelemetry 收集器统一格式
- 分成追踪、日志、指标三个流
- 存储和可视化用于监控

**API 网关用例**：
- 处理流量峰值（速率限制）
- 保护公共 API（认证授权）
- 减少往返（请求聚合）
- 支持多个客户端（版本控制）

**关键洞察**：
- 不同处理器设计用于不同工作负载
- GPU 通过大规模并行实现高性能
- API 网关集中处理横切关注点
- 分布式追踪对监控至关重要

理解这些概念对设计高性能系统至关重要。
