这是缓存速成课程系列的第三部分。

实现缓存系统时，不仅选择最佳缓存策略至关重要，还要解决可能出现的运维挑战。以下部分按类别深入探讨常见缓存问题，并提供如何有效应对这些挑战的见解。

## 缓存可靠性

缓存可靠性对维持稳定系统性能很重要。常见可靠性问题包括缓存雪崩、缓存击穿和缓存穿透。

缓存系统在通过维持高缓存命中率减少存储系统流量方面发挥关键作用。正常情况下，大多数读请求导致缓存命中，只有一小部分导致缓存未命中。按设计，缓存处理大部分流量，存储系统经历显著更少负载。

### 1. 缓存雪崩（Cache Avalanche）

缓存雪崩和缓存击穿是相关但不同的现象，可能发生在大规模系统中，导致显著性能下降甚至系统级中断。当多个或所有缓存条目同时或在短时间窗口内过期时发生缓存雪崩，导致对底层数据存储的请求突然激增。

### 2. 缓存击穿（Cache Stampede）

缓存击穿，也称为惊群效应，发生在大规模系统中当请求突然涌入使系统不堪重负，导致性能下降甚至系统级中断。这可能由于各种原因发生，如热门项目缓存未命中、用户流量突然激增，或维护后服务重启。

当缓存或缓存部分故障时，大量读请求导致缓存未命中，导致流量大量涌入存储系统。流量突然增加可能显著降低存储性能甚至导致系统崩溃。在此场景中，缓存故障充当触发器，结合缓存雪崩和缓存击穿元素，导致存储系统高压情况。

流量峰值，如美国电商网站黑色星期五销售等高流量事件期间，也可能触发这些现象。单个缓存节点可能因热门键需求增加而崩溃，导致这些键重新哈希到另一个节点，然后该节点也崩溃，最终导致整个缓存系统崩溃。

### 缓解策略

为维持大规模系统的稳定性和效率，实施策略减轻缓存雪崩和缓存击穿风险至关重要。有几种技术帮助预防和减轻这些事件的影响。

#### 缓存雪崩预防

- **交错过期**：通过结合基础 TTL 值与随机增量。此方法分散缓存条目的过期时间，减少许多项目同时过期的可能性。

- **一致性哈希**：可用于在多个缓存服务器间均匀分布缓存条目。此技术通过在剩余服务器间分担负载并防止单点故障减少缓存雪崩和缓存击穿的影响。

- **熔断器**：系统中实现熔断器可帮助防止缓存雪崩和缓存击穿升级为更严重问题。熔断器监控系统健康并在系统高压时通过停止对缓存和数据存储的过度请求防止过载。

- **请求速率限制和负载剥离**：可专门解决缓存击穿，通过控制请求处理速率并防止系统不堪重负。这些技术可按每用户、每客户端或系统范围应用以在高负载情况下维持稳定性。

### 3. 缓存穿透（Cache Penetration）

当应用尝试访问不存在的数据时发生缓存穿透，绕过缓存并导致潜在性能问题。当应用请求不存在的键时，它导致缓存未命中，随后向存储系统发送请求，存储系统也返回空结果。这种绕过缓存到达后备存储称为缓存穿透。如果发生大量不存在键请求，它可能影响存储层性能并使整体系统不稳定。

缓存穿透的常见例子是众多用户在短时间内尝试查询网站上不存在的订单。大网站通常实施保护措施缓解此问题。

#### 缓解方案

**方案 1：缓存占位符**
- 在缓存中存储占位符值表示不存在数据
- 后续对相同不存在数据的请求将命中缓存而不是数据存储
- 为这些占位符条目设置适当 TTL 以防止它们无限期占用缓存空间

虽然此方法简单有效，但如果应用频繁查询大量不存在键，它可能消耗大量缓存资源。

**方案 2：布隆过滤器**
- 布隆过滤器是节省空间、概率数据结构，测试元素是否属于集合
- 通过使用布隆过滤器，系统可以快速识别不存在数据，减少缓存穿透和不必要的数据存储请求
- 当记录添加到存储时，其键也记录在布隆过滤器中
- 获取记录时，应用首先检查键是否存在于布隆过滤器中
- 如果键不存在，它也不存在于缓存或存储中，应用可以直接返回空值
- 如果键存在于布隆过滤器中，应用继续像往常一样从缓存和存储读取键

布隆过滤器的挑战在于其容量限制。例如，10 亿个键、1% 假阳性率需要约 1.2 GB 容量。此解决方案最适合较小数据集。

## 热点 Key 问题

当单个键的高流量导致缓存节点承受过大流量压力时发生热点 key 问题，可能导致性能问题。

在分布式缓存系统中，键基于分片策略在不同节点间共享。每个键的流量可能差异很大，有些键经历异常高流量。如下图中所示，"key12"和"key15"等键的大量流量可能超过节点 1 的资源容量。

例如，当推文按其 ID 分布在缓存系统中时，其中少数推文迅速走红，相应缓存节点流量增加，可能超过其资源能力。热点 key 问题可能出现在各种场景中，如运营紧急情况、节日、体育比赛、闪购等。

### 解决方案

为解决此问题，需要两个步骤：

1. **识别热点 key**
   - 对于可预测事件（如重要节日或在线促销），可以预先评估潜在热点 key
   - 对于紧急情况，预先评估不可行。一个解决方案是进行实时流量分析以及时检测新出现的热点 key

2. **实施特殊措施**
   - **分散流量压力**：将"key12"拆分为"key12-1"、"key12-2"，直到"key12-N"，并将这 N 个键分布在多个节点上。当应用请求热点 key 时，它随机选择一个带后缀的键，使流量能够分散到许多节点上，防止单个节点过载
   - **实时扩展**：如果存在大量热点 key，实时监控可以使缓存系统快速扩展，减少流量影响
   - **本地缓存**：作为最后手段，应用可以将热点 key 存储在本地缓存中，减少对远程缓存系统的流量。此方法减轻缓存节点压力并帮助维持整体系统性能

## 大 Key 问题

当键的值大小非常大时，可能导致访问超时，产生大 key 问题。

大 key 问题的影响比最初看起来更严重：

- 频繁访问大 key 可能消耗大量网络带宽，导致查找性能缓慢
- 对大 key 的任何部分更新将导致整个值的修改，进一步导致性能问题
- 如果大 key 失效或从缓存驱逐，从存储重新加载它可能非常耗时，导致额外慢查询性能

## 译者总结

这篇文章涵盖了缓存系统常见问题的核心概念：

**缓存可靠性问题**：
| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 缓存雪崩 | 大量键同时过期 | 交错过期、一致性哈希、熔断器 |
| 缓存击穿 | 热门键失效、流量激增 | 速率限制、负载剥离 |
| 缓存穿透 | 访问不存在数据 | 缓存占位符、布隆过滤器 |

**热点 Key 问题**：
- 原因：单个键高流量压力
- 解决：识别 + 分散流量（拆分 key、实时扩展、本地缓存）

**大 Key 问题**：
- 影响：带宽消耗、更新慢、重新加载耗时
- 需要特殊处理策略

**关键洞察**：
- 缓存雪崩和击穿是相关但不同现象
- 布隆过滤器适合小数据集（10 亿键需 1.2GB）
- 热点 key 需要实时检测和分散
- 大 key 影响比表面更严重

理解这些概念对设计可靠缓存系统至关重要。
