## 5. 复制：弱一致性模型协议

现在我们已经查看了可以在越来越现实的受支持故障情况下强制执行单副本一致性的协议，让我们将注意力转向一旦我们放弃单副本一致性要求后打开的选项世界。

总的来说，很难提出一个单一维度来定义或表征允许副本发散的协议。大多数这样的协议都是高可用的，关键问题更多是最终用户是否发现保证、抽象和 API 对他们的目的有用，尽管当节点和/或网络故障发生时副本可能发散。

为什么弱一致性系统没有更流行？

正如我在引言中所述，我认为分布式编程的很大一部分是处理分布的两个后果的含义：

*   信息以光速传播
*   独立的事情独立故障

从信息传播速度限制得出的含义是，节点以不同的、独特的方式体验世界。单个节点上的计算很容易，因为一切都以可预测的全局总顺序发生。分布式系统上的计算很困难，因为没有全局总顺序。

在很长一段时间里（例如，几十年的研究），我们通过引入全局总顺序来解决这个问题。我已经讨论了许多通过创建秩序（以容错方式）来实现强一致性的方法，在没有自然发生的总顺序的地方。

当然，问题是强制执行秩序是昂贵的。这在大型互联网系统中尤其崩溃，系统需要保持可用。强制执行强一致性的系统不像分布式系统那样行为：它像单系统一样行为，这对分区期间的可用性不利。

此外，对于每个操作，通常必须联系大多数节点——而且通常不止一次，而是两次（正如你在 2PC 讨论中看到的）。这在需要地理分布以为全球用户群提供足够性能的系统中尤其痛苦。

所以默认像单系统一样行为可能不是理想的。

也许我们想要的是一个系统，我们可以在其中编写不使用昂贵协调的代码，但仍然返回"可用"的值。我们将允许不同的副本彼此发散——既为了保持效率也为了容忍分区——然后尝试以某种方式处理发散。

最终一致性表达了这个想法：节点可以在一段时间内彼此发散，但最终它们将同意该值。

在提供最终一致性的系统集中，有两种类型的系统设计：

**具有概率保证的最终一致性**。这种类型的系统可以在以后的某个时间点检测冲突写入，但不保证结果等同于某些正确的顺序执行。换句话说，冲突更新有时会导致用旧值覆盖新值，并且在正常运行期间（或分区期间）可以预期发生一些异常。

近年来，提供单副本一致性最有影响力的系统设计是 Amazon 的 Dynamo，我将它作为提供具有概率保证的最终一致性的系统示例进行讨论。

**具有强保证的最终一致性**。这种类型的系统保证结果收敛到等同于某些正确顺序执行的公共值。换句话说，这样的系统不产生任何异常结果；没有任何协调，你可以构建同一服务的副本，这些副本可以以任何模式通信并以任何顺序接收更新，并且只要它们都看到相同的信息，它们最终将同意最终结果。

CRDT（收敛复制数据类型）是保证收敛到相同值的数据类型，尽管有网络延迟、分区和消息重新排序。它们是可证明收敛的，但可以实现为 CRDT 的数据类型是有限的。

CALM（作为逻辑单调性的一致性）猜想是同一原则的替代表达：它将逻辑单调性与收敛等同。如果我们可以得出某物在逻辑上是单调的，那么无需协调安全地运行它也是安全的。汇合分析——特别是应用于 Bloom 编程语言——可用于指导程序员决定何时以及何地使用来自强一致性系统的协调技术，以及何时无需协调安全地执行。

## 调和不同的操作顺序

不强制执行单副本一致性的系统看起来像什么？让我们通过查看一些示例来使其更具体。

也许不强制执行单副本一致性的系统最明显的特征是它们允许副本彼此发散。这意味着没有严格定义的通信模式：副本可以彼此分离，但仍然保持可用并接受写入。

让我们想象一个三个副本的系统，每个副本都与其他副本分区。例如，副本可能位于不同的数据中心，并且由于某种原因无法通信。每个副本在分区期间保持可用，从某些客户端集接受读取和写入：

```
[Clients]   - > [A]

--- Partition ---

[Clients]   - > [B]

--- Partition ---

[Clients]   - > [C]
```

一段时间后，分区愈合，副本服务器交换信息。它们从不同的客户端接收了不同的更新并彼此发散，所以需要某种调和。我们希望发生的是所有副本收敛到相同的结果。

```
[A] \
     --> [merge]
[B] /     |
          |
[C] ----[merge]---> result
```

思考具有弱一致性保证的系统的另一种方式是想象一组客户端以某种顺序向两个副本发送消息。因为没有强制执行单个总顺序的协调协议，消息可以在两个副本上以不同的顺序传递：

```
[Clients]  --> [A]  1, 2, 3
[Clients]  --> [B]  2, 3, 1
```

这本质上就是我们需要协调协议的原因。例如，假设我们尝试连接字符串，消息 1、2 和 3 中的操作是：

```
1: { operation: concat("Hello ") }
2: { operation: concat("World") }
3: { operation: concat("!") }
```

那么，没有协调，A 将产生"Hello World!"，B 将产生"World!Hello "。

```
A: concat(concat(concat('", "Hello "), "World"), "!") = "Hello World!'
B: concat(concat(concat('", "World"), "!"), "Hello ") = "World!Hello '
```

这当然是不正确的。同样，我们希望发生的是副本收敛到相同的结果。

记住这两个示例，让我们先看看 Amazon 的 Dynamo 建立基线，然后讨论构建具有弱一致性保证的系统的许多新方法，如 CRDT 和 CALM 定理。

## Amazon 的 Dynamo

Amazon 的 Dynamo 系统设计（2007）可能是提供弱一致性保证但高可用性的最著名的系统。它是许多其他现实世界系统的基础，包括 LinkedIn 的 Voldemort、Facebook 的 Cassandra 和 Basho 的 Riak。

Dynamo 是一个最终一致的、高可用的键值存储。键值存储就像一个大型哈希表：客户端可以通过 `set(key, value)` 设置值，并使用 `get(key)` 按键检索它们。Dynamo 集群由 N 个对等节点组成；每个节点都有一组它负责存储的键。

Dynamo 优先考虑可用性而不是一致性；它不保证单副本一致性。相反，当写入值时，副本可能彼此发散；当读取键时，有一个读取调和阶段，尝试在将值返回给客户端之前调和副本之间的差异。

对于 Amazon 上的许多功能，避免中断比确保数据完全一致更重要，因为中断可能导致业务损失和信誉损失。此外，如果数据不是特别重要，那么弱一致性系统可以比传统 RDBMS 以更低的成本提供更好的性能和更高的可用性。

由于 Dynamo 是一个完整的系统设计，除了核心复制任务之外还有许多不同的部分需要查看。下图说明了其中一些任务；特别是，写入如何路由到节点并写入多个副本。

```
[ Client ]
    |
( Mapping keys to nodes )
    |
    V
[ Node A ]
    |     \
( Synchronous replication task: minimum durability )
    |        \
[ Node B]  [ Node C ]
    A
    |
( Conflict detection; asynchronous replication task:
  ensuring that partitioned / recovered nodes recover )
    |
    V
[ Node D]
```

在查看写入最初如何被接受之后，我们将查看如何检测冲突，以及异步副本同步任务。由于高可用性设计，此任务是必需的，其中节点可能暂时不可用（宕机或分区）。副本同步任务确保节点即使在故障后也能相当快速地赶上。

### 一致性哈希

无论是读取还是写入，需要发生的第一件事是我们需要定位数据应该生活在系统上的位置。这需要某种键到节点的映射。

在 Dynamo 中，键使用称为 [一致性哈希](https://github.com/mixu/vnodehash) 的哈希技术映射到节点（我不会详细讨论）。主要思想是，键可以通过客户端上的简单计算映射到负责它的一组节点。这意味着客户端可以定位键，而不必为每个键的位置查询系统；这节省了系统资源，因为哈希通常比执行远程过程调用更快。

### 部分仲裁

一旦我们知道键应该存储在哪里，我们需要做一些工作来持久化值。这是一个同步任务；我们立即将值写入多个节点的原因是提供更高水平的持久性（例如，保护节点立即故障）。

就像 Paxos 或 Raft 一样，Dynamo 使用仲裁进行复制。然而，Dynamo 的仲裁是松散（部分）仲裁而不是严格（多数）仲裁。

非正式地说，严格仲裁系统是一个仲裁系统，具有仲裁系统中的任何两个仲裁（集）重叠的属性。在接受之前要求多数投票支持更新保证只承认单个历史，因为每个多数仲裁必须至少在一个节点中重叠。这是 Paxos 例如依赖的属性。

部分仲裁没有该属性；这意味着不需要多数，并且仲裁的不同子集可能包含相同数据的不同版本。用户可以选择写入和读取的节点数：

*   用户可以选择某些数量的 W-of-N 节点需要写入成功；并且
*   用户可以指定在读取期间联系的节点数（R-of-N）。

`W` 和 `R` 指定需要参与写入或读取的节点数。写入更多节点使写入稍慢但增加了值不丢失的概率；从更多节点读取增加了读取值是最新的概率。

通常的建议是 `R + W > N`，因为这意味着读取和写入仲裁在一个节点中重叠——使得返回陈旧值的可能性更小。典型的配置是 `N = 3`（例如，每个值总共三个副本）；这意味着用户可以在以下之间选择：

```
R = 1, W = 3;
R = 2, W = 2 或
R = 3, W = 1
```

更一般地，再次假设 `R + W > N`：

*   `R = 1`，`W = N`：快速读取，慢速写入
*   `R = N`，`W = 1`：快速写入，慢速读取
*   `R = N/2` 和 `W = N/2 + 1`：对两者都有利

N 很少超过 3，因为保留那么多大量数据的副本变得昂贵！

正如我早些时候提到的，Dynamo 论文启发了许多其他类似的设计。它们都使用相同的部分仲裁基于复制方法，但 N、W 和 R 的默认值不同：

*   Basho 的 Riak（N = 3，R = 2，W = 2 默认）
*   Linkedin 的 Voldemort（N = 2 或 3，R = 1，W = 1 默认）
*   Apache 的 Cassandra（N = 3，R = 1，W = 1 默认）

还有另一个细节：发送读取或写入请求时，是要求所有 N 个节点响应（Riak），还是只满足最小数量的节点（例如 R 或 W；Voldemort）。"全部发送"方法更快且对延迟不太敏感（因为它只等待 N 个节点中最快的 R 或 W 个节点）但效率较低，而"发送到最小"方法对延迟更敏感（因为与单个节点通信的延迟将延迟操作）但也更高效（总体消息/连接更少）。

当读取和写入仲裁重叠时会发生什么，例如（`R + W > N`）？具体来说，通常声称这导致"强一致性"。

### R + W > N 与"强一致性"相同吗？

不。

这并非完全没有根据：`R + W > N` 的系统可以检测读取/写入冲突，因为任何读取仲裁和任何写入仲裁共享一个成员。例如，至少一个节点在两个仲裁中：

```
   1     2   N/2+1     N/2+2    N
  [...] [R]  [R + W]   [W]    [...]
```

这保证先前的写入将被后续读取看到。然而，这仅在 N 中的节点从不更改时才成立。因此，Dynamo 不符合条件，因为在 Dynamo 中，如果节点故障，集群成员可以更改。

Dynamo 设计为始终可写入。它有一个机制，当原始服务器宕机时，通过将不同的、不相关的服务器添加到负责某些键的节点集中来处理节点故障。这意味着仲裁不再保证总是重叠。即使 `R = W = N` 也不符合，因为虽然仲裁大小等于 N，但这些仲裁中的节点可以在故障期间更改。具体来说，在分区期间，如果无法到达足够数量的节点，Dynamo 将从不相关但可访问的节点向仲裁添加新节点。

此外，Dynamo 不会以强制执行强一致性模型的系统的方式处理分区：即，允许在分区的两侧写入，这意味着至少在某些时间内，系统不作为单副本行为。所以称 `R + W > N` 为"强一致"是误导性的；保证仅仅是概率的——这不是强一致性所指的。

### 冲突检测和读取修复

允许副本发散的系统必须有办法最终调和两个不同的值。正如在部分仲裁方法期间简要提到的，一种方法是在读取时检测冲突，然后应用某些冲突解决方法。但这是如何完成的？

一般来说，这是通过用某些元数据补充数据来跟踪数据的因果历史来完成的。客户端从系统读取数据时必须保留元数据信息，并且在写入数据库时必须返回元数据值。

我们已经遇到了一种这样做的方法：向量时钟可用于表示值的历史。事实上，这是原始 Dynamo 设计用于检测冲突的方法。

然而，使用向量时钟不是唯一的替代方案。如果你查看许多实际系统设计，你可以通过查看它们跟踪的元数据来推断出很多关于它们如何工作的信息。

**无元数据**。当系统不跟踪元数据，并且只返回值（例如，通过客户端 API）时，它不能对并发写入做任何特别的事情。常见的规则是最后写入者获胜：换句话说，如果两个写入者同时写入，则只保留来自最慢写入者的值。

**时间戳**。名义上，具有较高时间戳值的值获胜。然而，如果时间没有仔细同步，可能会发生许多奇怪的事情，来自具有故障或快速时钟的系统的旧数据覆盖较新的值。Facebook 的 Cassandra 是使用时间戳而不是向量时钟的 Dynamo 变体。

**版本号**。版本号可能避免使用 timestamps 相关的一些问题。注意，当可能存在多个历史时，可以准确跟踪因果关系的最小机制是向量时钟，而不是版本号。

**向量时钟**。使用向量时钟，可以检测并发和过时的更新。然后可以进行读取修复，尽管在某些情况下（并发更改）我们需要要求客户端选择一个值。这是因为如果更改是并发的，并且我们对数据一无所知（如简单键值存储的情况），那么询问比任意丢弃数据更好。

读取值时，客户端联系 `R` 个 `N` 节点，并要求它们提供键的最新值。它接受所有响应，丢弃严格较旧的值（使用向量时钟值检测这一点）。如果只有一个唯一的向量时钟 + 值对，则返回该对。如果有多个向量时钟 + 值对已并发编辑（例如，不可比较），则返回所有这些值。

从上面可以看出，读取修复可能返回多个值。这意味着客户端/应用程序开发人员必须偶尔通过根据某些用例特定标准选择值来处理这些情况。

此外，实际向量时钟系统的一个关键组成部分是时钟不能被允许永远增长——所以需要有一个过程来偶尔以安全的方式垃圾回收时钟，以平衡容错与存储要求。

### 副本同步：gossip 和 Merkle 树

鉴于 Dynamo 系统设计容忍节点故障和网络分区，它需要一种方法来处理节点在被分区后重新加入集群，或者当故障节点被替换或部分恢复时。

副本同步用于在故障后使节点最新，并用于定期使副本彼此同步。

Gossip 是一种同步副本的概率技术。通信模式（例如，哪个节点联系哪个节点）不是预先确定的。相反，节点有一定的概率 `p` 尝试彼此同步。每隔 `t` 秒，每个节点选择一个节点进行通信。这提供了除同步任务（例如，部分仲裁写入）之外的额外机制，使副本最新。

Gossip 是可扩展的，没有单点故障，但只能提供概率保证。

为了使副本同步期间的信息交换高效，Dynamo 使用称为 Merkle 树的技术，我不会详细讨论。主要思想是，数据存储可以在多个不同的粒度级别进行哈希：表示整个内容的哈希、一半键、四分之一键等等。

通过维护这种相当细粒度的哈希，节点可以比朴素技术更有效地比较它们的数据存储内容。一旦节点识别出哪些键具有不同的值，它们交换必要的信息以使副本最新。

### 实践中的 Dynamo：概率有界陈旧性（PBS）

这几乎涵盖了 Dynamo 系统设计：

*   一致性哈希确定键放置
*   部分仲裁用于读取和写入
*   通过向量时钟进行冲突检测和读取修复
*   gossip 用于副本同步

我们如何表征这种系统的行为？Bailis 等人（2012）的一篇相当新的论文描述了一种称为 [PBS](http://pbs.cs.berkeley.edu/)（概率有界陈旧性）的方法，使用模拟和从现实世界系统收集的数据来表征这种系统的预期行为。

PBS 通过使用关于反熵（gossip）速率、网络延迟和局部处理延迟的信息来估计不一致的程度，以估计读取的预期一致性水平。它已在 Cassandra 中实现，其中定时信息被搭载在其他消息上，并且基于蒙特卡罗模拟中此信息的样本计算估计。

根据该论文，在正常运行期间，最终一致的数据存储通常更快，并且可以在几十或几百毫秒内读取一致的状态。下表说明了在 LinkedIn（SSD 和 15k RPM 磁盘）和 Yammer 的经验定时数据上，给定不同的 `R` 和 `W` 设置，从 99.9% 一致读取概率所需的时间量：

![来自 PBS 论文](https://book.mixu.net/distsys/images/pbs.png)

例如，在 Yammer 情况下，从 `R=1`，`W=1` 到 `R=2`，`W=1` 将不一致窗口从 1352 ms 减少到 202 ms——同时保持读取延迟低于最快的严格仲裁（`R=3`，`W=1`；219.27 ms）（32.6 ms）。

有关更多详细信息，请查看 [PBS 网站](http://pbs.cs.berkeley.edu/) 和相关论文。

## 无序编程

让我们回顾一下我们想要解决的情况示例。第一个场景由分区后面的三个不同服务器组成；分区愈合后，我们希望服务器收敛到相同的值。Amazon 的 Dynamo 通过从 `N` 个节点中的 `R` 个节点读取然后执行读取调和来实现这一点。

在第二个示例中，我们考虑了一个更具体的操作：字符串连接。事实证明，没有已知的技术可以使字符串连接解析为相同的值，而不对操作强加顺序（例如，没有昂贵的协调）。然而，有些操作可以以任何顺序安全地应用，而简单寄存器将无法做到这一点。正如 Pat Helland 所写：

> ... 以操作为中心的工作可以使可交换（具有正确的操作和正确的语义），而简单的 READ/WRITE 语义不适合可交换性。

例如，考虑一个系统，以两种不同的方式实现一个简单的会计系统，具有 `debit` 和 `credit` 操作：

*   使用具有 `read` 和 `write` 操作的寄存器，以及
*   使用具有本机 `debit` 和 `credit` 操作的整数数据类型

后一种实现更了解数据类型的内部，因此它可以保留操作的意图，尽管操作被重新排序。借记或贷记可以以任何顺序应用，最终结果相同：

```
100 + credit(10) + credit(20) = 130 且
100 + credit(20) + credit(10) = 130
```

然而，写入固定值不能以任何顺序完成：如果写入被重新排序，其中一个写入将覆盖另一个：

```
100 + write(110) + write(130) = 130 但
100 + write(130) + write(110) = 110
```

让我们从本章开头的示例开始，但使用不同的操作。在此场景中，客户端向两个节点发送消息，这两个节点以不同的顺序查看操作：

```
[Clients]  --> [A]  1, 2, 3
[Clients]  --> [B]  2, 3, 1
```

不是字符串连接，假设我们正在寻找一组整数的最大值（例如，MAX()）。消息 1、2 和 3 是：

```
1: { operation: max(previous, 3) }
2: { operation: max(previous, 5) }
3: { operation: max(previous, 7) }
```

那么，没有协调，A 和 B 都将收敛到 7，例如：

```
A: max(max(max(0, 3), 5), 7) = 7
B: max(max(max(0, 5), 7), 3) = 7
```

在这两种情况下，两个副本以不同的顺序查看更新，但我们能够以相同结果的方式合并结果，尽管顺序如何。结果在这两种情况下都收敛到相同的答案，因为我们使用的合并过程（`max`）。

可能不可能为所有数据类型编写合并过程。在 Dynamo 中，值是二进制 blob，所以能做的最好的事情是公开它并要求应用程序处理每个冲突。

然而，如果我们知道数据是更具体的类型，处理这类冲突就成为可能。CRDT 是设计为提供数据类型的结构，只要它们看到相同的操作集（以任何顺序），这些数据类型将始终收敛。

## CRDT：收敛复制数据类型

CRDT（收敛复制数据类型）利用关于特定数据类型上特定操作的交换性和结合性的知识。

为了使一组操作在副本只偶尔通信的环境中收敛到相同的值，操作需要与顺序无关且对（消息）重复/重新传递不敏感。因此，它们的操作需要是：

*   **结合的**（`a+(b+c)=(a+b)+c`），因此分组不重要
*   **交换的**（`a+b=b+a`），因此应用顺序不重要
*   **幂等的**（`a+a=a`），因此重复不重要

事实证明，这些结构在数学中已经已知；它们被称为 join 或 meet [半格](https://en.wikipedia.org/wiki/Semilattice)。

[格](https://en.wikipedia.org/wiki/Lattice_%28order%29) 是一个偏序集，具有不同的顶部（最小上界）和不同的底部（最大下界）。半格就像格，但只有一个不同的顶部或底部。join 半格是具有不同顶部（最小上界）的半格，meet 半格是具有不同底部（最大下界）的半格。

任何可以表示为半格的数据类型都可以实现为保证收敛的数据结构。例如，计算一组值的 `max()` 将始终返回相同的结果，无论接收值的顺序如何，只要最终接收所有值，因为 `max()` 操作是结合的、交换的和幂等的。

例如，这里有两个格：一个为集合绘制，其中合并运算符是 `union(items)`，另一个为严格递增的整数计数器绘制，其中合并运算符是 `max(values)`：

```
   { a, b, c }              7
  /      |    \            /  \
{a, b} {b,c} {a,c}        5    7
  |  \  /  | /           /   |  \
  {a} {b} {c}            3   5   7
```

对于可以表示为半格的数据类型，你可以让副本以任何模式通信并以任何顺序接收更新，并且只要它们都看到相同的信息，它们最终将同意最终结果。这是一个强大的属性，只要前提条件成立就可以保证。

然而，将数据类型表示为半格通常需要某种程度的解释。许多数据类型具有实际上与顺序无关的操作。例如，向集合添加项目是结合的、交换的和幂等的。然而，如果我们也允许从集合中删除项目，那么我们需要某种方法来解决冲突操作，例如 `add(A)` 和 `remove(A)`。如果本地副本从未添加它，删除元素意味着什么？这种分辨率必须以与顺序无关的方式指定，并且有几种不同的选择具有不同的权衡。

这意味着几个熟悉的数据类型作为 CRDT 具有更专门的实现，这些实现做出不同的权衡以以与顺序无关的方式解决冲突。与简单处理寄存器的键值存储不同（例如，从系统的角度来看，值是不透明的 blob），使用 CRDT 的人必须使用正确的数据类型来避免异常。

一些指定为 CRDT 的不同数据类型的示例包括：

*   **计数器**
    *   只增计数器（merge = max(values); payload = 单个整数）
    *   正负计数器（由两个增长计数器组成，一个用于增量，另一个用于减量）

*   **寄存器**
    *   最后写入获胜寄存器（时间戳或版本号；merge = max(ts); payload = blob）
    *   多值寄存器（向量时钟；merge = 两者都取）

*   **集合**
    *   只增集合（merge = union(items); payload = 集合；无删除）
    *   两阶段集合（由两个集合组成，一个用于添加，另一个用于删除；元素可以添加一次并删除一次）
    *   唯一集合（两阶段集合的优化版本）
    *   最后写入获胜集合（merge = max(ts); payload = 集合）
    *   正负集合（每个集合项由一个 PN 计数器组成）
    *   观察 - 删除集合

*   **图和文本序列**（参见论文）

为确保无异常操作，你需要为你的特定应用程序找到正确的数据类型——例如，如果你知道你只会删除项目一次，那么两阶段集合有效；如果你只会向集合添加项目而从不删除它们，那么只增集合有效。

并非所有数据结构都有已知的 CRDT 实现，但在 Shapiro 等人最近的（2011）[调查论文](http://hal.inria.fr/docs/00/55/55/88/PDF/techreport.pdf) 中有布尔值、计数器、集合、寄存器和图的 CRDT 实现。

有趣的是，寄存器实现直接对应于键值存储使用的实现：最后写入获胜寄存器使用时间戳或某些等效物，并简单地收敛到最大时间戳值；多值寄存器对应于 Dynamo 保留、公开和调和并发更改的策略。有关详细信息，我建议你查看本章进一步阅读部分中的论文。

## CALM 定理

CRDT 数据结构基于对可表示为半格的数据结构是收敛的认识。但编程不仅仅是发展状态，除非你只是实现数据存储。

显然，顺序独立性是任何收敛计算的重要属性：如果接收数据项的顺序影响计算结果，那么无法在不保证顺序的情况下执行计算。

然而，有许多编程模型中语句的顺序不起重要作用。例如，在 [MapReduce 模型](https://en.wikipedia.org/wiki/MapReduce) 中，Map 和 Reduce 任务都被指定为需要在数据集上运行的无状态元组处理任务。关于数据如何以及以何种顺序路由到任务的具体决定没有明确指定，而是批处理作业调度器负责调度任务在集群上运行。

类似地，在 SQL 中，人们指定查询，但不指定如何执行查询。查询只是任务的声明性描述，查询优化器的工作是找出执行查询的有效方法（跨多台机器、数据库和表）。

当然，这些编程模型不如通用编程语言那样宽容。MapReduce 任务需要可表示为无环数据流程序中的无状态任务；SQL 语句可以执行相当复杂的计算，但许多事情很难用它表达。

然而，从这两个例子应该清楚的是，有许多种数据处理任务适合以声明式语言表达，其中执行顺序没有明确指定。表达所需结果同时将语句的确切顺序留给优化器决定的编程模型通常具有与顺序无关的语义。这意味着这样的程序可能无需协调即可执行，因为它们依赖于它们接收的输入，但不一定依赖于接收输入的具体顺序。

关键是这样的程序_可能_安全地无需协调即可执行。如果没有明确的规则来表征什么可以安全地无需协调即可执行，什么不能，我们就无法在保持结果正确的同时实现程序。

这就是 CALM 定理的内容。CALM 定理基于对逻辑单调性和有用形式的最终一致性（例如，汇合/收敛）之间联系的认识。它指出逻辑单调的程序保证最终一致。

那么，如果我们知道某些计算在逻辑上是单调的，那么我们也知道它无需协调即可安全地执行。

为了更好地理解这一点，我们需要对比单调逻辑（或单调计算）与 [非单调逻辑](http://plato.stanford.edu/entries/logic-nonmonotonic/)（或非单调计算）。

**单调性** 如果句子 `φ` 是一组前提 `Γ` 的结果，那么它也可以从任何扩展 `Γ` 的前提集 `Δ` 推断出来

大多数标准逻辑框架是单调的：在诸如一阶逻辑之类的框架内做出的任何推断，一旦演绎有效，就不能被新信息 invalidated。非单调逻辑是该属性不成立的系统——换句话说，如果通过学习新知识可以使某些结论无效。

在人工智能界，非单调逻辑与 [可废止推理](http://plato.stanford.edu/entries/reasoning-defeasible/) 相关联——推理，其中使用部分信息做出的断言可以被新知识 invalidated。例如，如果我们了解到 Tweety 是一只鸟，我们会假设 Tweety 会飞；但如果我们后来了解到 Tweety 是一只企鹅，那么我们将不得不修改我们的结论。

单调性涉及前提（或关于世界的事实）和结论（或关于世界的断言）之间的关系。在单调逻辑内，我们知道我们的结果是无需撤回的：[单调](https://en.wikipedia.org/wiki/Monotonicity_of_entailment) 计算不需要重新计算或协调；答案随着时间的推移变得更加准确。一旦我们知道 Tweety 是一只鸟（并且我们使用单调逻辑进行推理），我们可以安全地得出结论 Tweety 会飞，我们学到的任何东西都不能 invalidated 这个结论。

虽然任何产生面向人类结果的计算都可以解释为关于世界的断言（例如，"foo"的值是"bar"），但很难确定基于冯·诺依曼机的编程模型中的计算是否是单调的，因为事实和断言之间的关系以及这些关系是否是单调的并不完全清楚。

然而，有许多编程模型可以确定单调性。特别是，[关系代数](https://en.wikipedia.org/wiki/Relational_algebra)（例如，SQL 的理论基础）和 [Datalog](https://en.wikipedia.org/wiki/Datalog) 提供了具有充分理解解释的高度表达语言。

基本 Datalog 和关系代数（即使有递归）都被认为是单调的。更具体地说，使用某些基本运算符集表达的计算被认为是单调的（选择、投影、自然连接、叉积、并集和无否定的递归 Datalog），非单调性通过使用更高级的运算符引入（否定、集差、除法、全称量化、聚合）。

这意味着在这些系统中使用大量运算符（例如，map、filter、join、union、intersection）表达的计算在逻辑上是单调的；使用这些运算符的任何计算也是单调的，因此无需协调即可安全地运行。另一方面，使用否定和聚合的表达式无需协调即可安全地运行。

重要的是要认识到非单调性和在分布式系统中执行昂贵的操作之间的联系。具体来说，_分布式聚合_ 和_协调协议_ 都可以被认为是一种否定形式。正如 Joe Hellerstein [所写](http://www.eecs.berkeley.edu/Pubs/TechRpts/2010/EECS-2010-90.pdf)：

> 为了在分布式设置中建立否定谓词的真实性，评估策略必须开始"计数到 0"以确定空性，并等待分布式计数过程明确终止。聚合是这个想法的概括。

并且：

> 这个想法也可以从另一个方向看到。协调协议本身就是聚合，因为它们需要投票：两阶段提交需要一致投票，Paxos 共识需要多数投票，拜占庭协议需要 2/3 多数。等待需要计数。

如果，那么我们可以以可以测试单调性的方式表达我们的计算，那么我们可以执行全程序静态分析，检测程序的哪些部分最终一致且无需协调即可安全地运行（单调部分）——哪些部分不是（非单调部分）。

注意，这需要一种不同类型的语言，因为这些推断很难为传统编程语言做出，其中序列、选择和迭代是核心。这就是 Bloom 语言设计的原因。

## 非单调性有什么好处？

单调性和非单调性之间的区别很有趣。例如，将两个数字相加是单调的，但计算包含数字的两个节点上的聚合不是。有什么区别？其中一个是计算（将两个数字相加），而另一个是断言（计算聚合）。

计算与断言有何不同？让我们考虑查询"pizza 是蔬菜吗？"。要回答这个问题，我们需要深入核心：什么时候可以推断某物是（或不是）真的？

有几个可接受的答案，每个答案对应于关于我们拥有的信息以及我们应该如何对其采取行动的不同假设集——我们在不同的上下文中接受了不同的答案。

在日常推理中，我们做出所谓的 [开放世界假设](https://en.wikipedia.org/wiki/Open_world_assumption)：我们假设我们不知道一切，因此不能从缺乏知识中得出结论。也就是说，任何句子可能是真的、假的或未知的。

```
                                OWA +             |  OWA +
                                Monotonic logic   |  Non-monotonic logic
Can derive P(true)      |   Can assert P(true)    |  Cannot assert P(true)
Can derive P(false)     |   Can assert P(false)   |  Cannot assert P(true)
Cannot derive P(true)   |   Unknown               |  Unknown
or P(false)
```

当做出开放世界假设时，我们只能安全地断言我们可以从已知内容中推断出的内容。我们关于世界的信息被假定为不完整的。

让我们首先看看我们知道我们的推理是单调的情况。在这种情况下，我们拥有的任何（可能不完整的）知识都不能被学习新知识 invalidated。所以如果我们可以从某些演绎中推断出句子是真的，例如"含有两汤匙番茄酱的东西是蔬菜"和"pizza 含有两汤匙番茄酱"，那么我们可以得出结论"pizza 是蔬菜"。如果我们能推断出句子是假的，也是一样。

然而，如果我们不能推断出任何东西——例如，我们拥有的知识集包含客户信息，而没有关于 pizza 或蔬菜的信息——那么在开放世界假设下，我们必须说我们不能得出任何结论。

对于非单调知识，我们现在知道的任何事情都可能被 invalidated。因此，我们不能安全地得出任何结论，即使我们可以从目前知道的内容中推断出真或假。

然而，在数据库上下文中，以及在许多计算机科学应用中，我们更喜欢做出更明确的结论。这意味着假设所谓的 [封闭世界假设](https://en.wikipedia.org/wiki/Closed_world_assumption)：任何不能被证明为真的东西都是假的。这意味着不需要明确声明虚假。换句话说，我们拥有的事实数据库被假定为完整的（最小的），因此任何不在其中的东西都可以假定为假的。

例如，在 CWA 下，如果我们的数据库没有旧金山和赫尔辛基之间的航班条目，那么我们可以安全地得出不存在这样的航班。

我们需要一件事才能做出明确的断言：[逻辑限制](https://en.wikipedia.org/wiki/Circumscription_%28logic%29)。限制是一个形式化的猜测量规则。域限制猜测已知实体是所有存在的实体。我们需要能够假设已知实体是所有存在的实体，才能得出明确的结论。

```
                                CWA +             |  CWA +
                                Circumscription + |  Circumscription +
                                Monotonic logic   |  Non-monotonic logic
Can derive P(true)      |   Can assert P(true)    |  Can assert P(true)
Can derive P(false)     |   Can assert P(false)   |  Can assert P(false)
Cannot derive P(true)   |   Can assert P(false)   |  Can assert P(false)
or P(false)
```

特别是，非单调推断需要这个假设。只有当我们假设我们拥有完整的信息时，我们才能做出自信的断言，因为否则额外的信息可能会 invalidated 我们的断言。

这在实践中意味着什么？首先，单调逻辑一旦可以推断出句子是真的（或假的）就可以得出明确的结论。其次，非单调逻辑需要一个额外的假设：已知实体是所有存在的实体。

那么为什么两个表面上等效的操作不同？为什么将两个数字相加是单调的，但计算两个节点上的聚合不是？因为聚合不仅计算总和，还断言它已经看到了所有的值。保证这一点的唯一方法是跨节点协调，并确保执行计算的节点确实看到了系统中的所有值。

因此，为了处理非单调性，人们需要使用分布式协调来确保只在知道所有信息后才做出断言，或者做出结论可以在以后 invalidated 的警告断言。

处理非单调性对于表达性原因很重要。这归结为能够表达非单调事物；例如，可以说某列的总和是 X 很好。系统必须检测到这种计算需要全局协调边界，以确保我们已经看到了所有实体。

纯单调系统是罕见的。似乎大多数应用程序即使在数据不完整的情况下也在封闭世界假设下运行，我们人类对此很好。当数据库告诉你旧金山和赫尔辛基之间没有直飞航班时，你可能会将其视为"根据这个数据库，没有直飞航班"，但你不会排除现实中这样的航班可能仍然存在的可能性。

实际上，只有当副本可以发散时（例如，在分区期间或由于正常运行期间的延迟），这个问题才变得有趣。然后需要更具体的考虑：答案是否仅基于当前节点，还是系统的整体。

此外，由于非单调性是由做出断言引起的，似乎许多计算可以进行很长时间，并且只在某些结果或断言传递给第三方系统或最终用户时才应用协调。当然，如果这些读取和写入只是长时间运行计算的一部分，则系统内的每个单个读取和写入操作都不需要强制执行总顺序。

## Bloom 语言

[Bloom 语言](http://www.bloom-lang.net/) 是一种设计为利用 CALM 定理的语言。它是一个 Ruby DSL，其形式基础是称为 Dedalus 的时序逻辑编程语言。

在 Bloom 中，每个节点都有一个由集合和格组成的数据库。程序表示为与集合（事实集）和格（CRDT）交互的无序语句集。语句默认与顺序无关，但人们也可以编写非单调函数。

请查看 [Bloom 网站](http://www.bloom-lang.net/) 和 [教程](https://github.com/bloom-lang/bud/tree/master/docs) 以了解有关 Bloom 的更多信息。
