Netflix 于 1997 年开始作为基于邮件的 DVD 租赁业务。

Marc Randolph 和 Reed Hastings 在加利福尼亚州办公室和家之间拼车时想到了 Netflix 的想法。

Hastings 钦佩亚马逊，想通过在互联网上销售大量便携式商品来模仿他们的成功。大约在同一时间，DVD 在美国推出，他们测试了通过邮件销售或租赁 DVD 的概念。

快进到 2024 年，Netflix 已发展成为拥有来自世界各地超过 2.6 亿用户的视频流媒体服务。它的影响如此巨大，以至于"Netflix 宕机"通常被视为紧急情况。

为了支持这个惊人的增长故事，Netflix 必须在多个维度上扩展其架构。

在本文中，我们试图揭开他们所面临的一些最重大扩展挑战以及如何克服这些挑战的面纱。

## 单体架构时代

像任何希望快速进入竞争市场的初创公司一样，Netflix 开始时是一个单体应用。

下图显示了很久以前他们的架构是什么样子的。

![早期 Netflix 架构](01.webp)

该应用由一个单一的可部署单元和一个单体数据库（Oracle）组成。正如你所注意到的，数据库是一个可能的单点故障。

这种可能性在 2008 年 8 月变成了现实。

由于重大数据库损坏问题，Netflix 无法向客户运送任何 DVD，持续了 3 天。突然间，他们必须摆脱容易受到单点故障影响的垂直扩展架构变得显而易见。

作为回应，他们做出了两个重要决定：
- **将所有数据迁移到 AWS 云平台**
- **将系统演变为基于微服务的架构**

## 迁移到 AWS

迁移到 AWS 是一个关键决定。

当 Netflix 在 2007 年推出时，EC2 刚刚起步，他们当时无法利用它。因此，他们建立了两个彼此相邻的数据中心。

然而，建立数据中心有很多工作。你必须订购设备，等待设备到达并安装它。在你完成之前，你已经再次耗尽容量，需要再次经历整个周期。

为了打破这个循环，Netflix 采用了垂直扩展策略，导致他们的早期系统架构被建模为单体应用。

然而，我们之前谈到的中断给 Netflix 上了一个关键教训——建立数据中心不是他们的核心能力。

他们的核心能力是向订阅者交付视频，如果他们能更好地交付视频会更好。这促使他们迁移到 AWS，采用可以消除单点故障的设计方法。

对于当时来说，这是一个巨大的决定，Netflix 采用了一些基本原则来指导他们度过这个变化：

### 开源原则
- 首先，尽可能使用或贡献开源技术
- 只从头构建你绝对必须的东西

### 无状态服务
- 服务应以无状态方式构建，持久化或缓存层除外
- 无粘性会话
- 采用混沌测试来证明实例宕机不会影响更广泛的系统

### 水平扩展
- 水平扩展为你提供更长的可扩展性跑道
- 优先选择水平扩展而不是垂直扩展

### 冗余
- 任何事物都要制作多个副本。例如，副本数据库和多个服务实例
- 通过隔离工作负载减少任何问题的爆炸半径

### 混沌工程
- 系统的破坏性测试应该是持续活动
- 采用 Chaos Monkey 等工具在大规模上进行此类测试

这些指导原则充当了 Netflix 为构建可根据需求扩展的架构而采取的每个转型项目的北极星。

## Netflix 架构三部分组成

整体 Netflix 架构分为三部分：
- **客户端**
- **后端**
- **内容分发网络（CDN）**

![Netflix 架构三部分](02.webp)

### 1. 客户端
客户端是你移动设备上的 Netflix 应用、计算机上的网站，甚至是智能电视上的应用。它包括用户可以浏览和流式传输 Netflix 视频的任何设备。Netflix 控制着每个设备的每个客户端。

### 2. 后端
后端是控制用户点击播放之前发生的一切的应用部分。它由在 AWS 上运行的多个服务组成，负责各种功能，如用户注册、准备传入视频、计费等。后端暴露多个 API，客户端利用这些 API 提供无缝的用户体验。

### 3. 内容分发网络（Open Connect）
第三部分是内容分发网络，也称为 Open Connect。它在世界各地的不同位置存储 Netflix 视频。当用户播放视频时，它从 Open Connect 流式传输并显示在客户端上。

重要的是要注意，Netflix 控制着所有三个领域，从而实现对其栈的完全垂直集成。

## 关键扩展领域

Netflix 要想成功，必须扩展的一些关键领域如下：
- 内容分发网络（CDN）
- Netflix 边缘
- API
- 带缓存的后端服务
- 授权
- 会员资格

让我们更详细地看看每个领域。

### CDN 扩展

想象一下你正在新加坡观看视频，而视频是从波特兰流式传输的。这是一个巨大的地理距离，分成许多网络跳数。在这种设置中必然会出现延迟问题，导致用户体验较差。

如果视频内容更靠近观看它的人，观看体验会更好。

这是 Netflix 使用 CDN 背后的基本思想。

通过在世界各地的位置存储副本来尽可能将视频靠近用户。当用户想要观看视频时，从最近的节点流式传输。

每个存储视频内容的位置称为 PoP 或存在点。它是一个提供互联网访问的物理位置，由服务器、路由器和其他网络设备组成。

然而，Netflix 花了多次迭代才将 CDN 扩展到合适的水平。

#### 第一阶段：自建 CDN（2007）
Netflix 在 2007 年推出了流媒体服务。

当时，它拥有超过 3500 万会员，遍布 50 个国家，每月流式传输超过 10 亿小时的视频。

为了支持这种使用，Netflix 在美国境内的五个不同地点构建了自己的 CDN。每个位置都包含所有内容。

#### 第二阶段：第三方 CDN（2009）
2009 年，Netflix 开始使用第三方 CDN。

原因是第三方 CDN 成本正在下降，Netflix 投入大量时间和精力构建自己的 CDN 没有意义。正如我们所看到的，他们在运行自己的数据中心方面 struggles 很多。

转向第三方解决方案也给了他们时间处理其他更高优先级的项目。然而，Netflix 确实花了很多时间和精力开发更智能的客户端应用以适应不断变化的网络条件。

例如，他们开发了切换到不同 CDN 以获得更好结果的技术。此类创新使他们能够在面对错误和过载网络的情况下为用户提供最高质量的体验。

#### 第三阶段：Open Connect（2012）
大约在 2011 年，Netflix 意识到他们的运营规模需要专用 CDN 来最大化网络效率和观看体验。

流媒体业务现在是主要的收入来源，视频分发是核心能力。如果他们能以极高质量做到这一点，它可以变成巨大的竞争优势。

因此，2012 年，Netflix 推出了自己的 CDN，称为 Open Connect。

为了获得最佳性能，他们开发了自己的视频存储计算机系统，称为 Open Connect Appliances 或 OCA。

![OCA 安装](03.webp 失败)

OCA 安装是多个 OCA 服务器的集群。每个 OCA 是高度优化的快速服务器，用于交付大文件。它们装满了许多硬盘或闪存驱动器用于存储视频。

Open Connect CDN 的推出对 Netflix 有很多优势：
- **更具可扩展性**：在世界各地提供服务方面
- **更高质量**：因为他们现在可以控制从转码、CDN 到设备上客户端的整个视频路径
- **更便宜**：与第三方 CDN 相比

### 边缘扩展

扩展拼图的下一个关键部分是边缘。

边缘是系统中靠近客户端的部分。例如，在 DNS 和数据库之间，DNS 更靠近客户端，可以被认为是更边缘的。把它看作是程度而不是固定值。

边缘是来自各种请求的数据进入服务域的地方。由于这是请求量最高的地方，因此扩展边缘至关重要。

Netflix 边缘在扩展方面经历了多个阶段。

下图显示了 Netflix 架构在最初几天的样子。

![早期三层层架构](04.webp)

正如你所看到的，这是一个典型的三层层架构。

有一个客户端、一个 API 和 API 对话的数据库。API 应用名为 NCCP（Netflix 内容控制协议），它是唯一暴露给客户端的应用。所有关注点都放入这个应用中。

负载均衡器终止 TLS 并将纯流量发送到应用。此外，DNS 配置非常简单。想法是客户端应该能够找到并到达 Netflix 服务器。

这样的设计是由当时的业务需求决定的。他们有钱但不多。重要的是不要使事情过于复杂并优化上市时间。

随着客户群的增长，添加了更多功能。随着更多功能，公司开始赚更多钱。

此时，维护工程速度对他们来说很重要。这意味着将单体应用分解为微服务。功能从 NCCP 应用中取出，作为具有单独数据的单独应用开发。

然而，服务之间编排的逻辑仍在 API 内。来自客户端的传入请求击中 API，API 按正确顺序调用底层微服务。

## 译者总结

这篇文章介绍了 Netflix 的架构演进历程：

**发展历程**：
- 1997 年：DVD 租赁业务开始
- 2007 年：推出流媒体服务
- 2008 年：数据库故障，决定迁移 AWS
- 2012 年：推出 Open Connect CDN
- 2024 年：2.6 亿用户

**关键决策**：
1. 迁移到 AWS（消除单点故障）
2. 微服务架构（提高可扩展性）
3. 自建 Open Connect CDN（控制质量和成本）

**指导原则**：
- 优先使用开源
- 无状态服务
- 水平扩展优于垂直扩展
- 冗余设计
- 混沌工程测试

**架构三部分**：
1. 客户端（所有设备）
2. 后端（AWS 上的微服务）
3. CDN（Open Connect 全球分发）

**CDN 演进**：
- 2007：自建 5 个数据中心
- 2009：第三方 CDN
- 2012：Open Connect（自建专用 CDN）

Netflix 的扩展经验为现代云原生架构提供了宝贵参考。
