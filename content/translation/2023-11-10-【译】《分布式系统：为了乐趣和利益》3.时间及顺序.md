## 3. 时间及顺序

什么是顺序，为什么它很重要？

你什么意思"什么是顺序"？

我是说，为什么我们一开始就如此痴迷于顺序？为什么我们在乎 A 是否发生在 B 之前？为什么我们不在乎一些其他属性，比如"颜色"？

嗯，我疯狂的朋友，让我们回到分布式系统的定义来回答这个问题。

正如你可能记得的，我将分布式编程描述为使用多台计算机解决你可以在单台计算机上解决的同一问题的艺术。

事实上，这是对顺序痴迷的核心。任何一次只能做一件事的系统都会创建一个操作的总顺序。就像人们穿过一扇门，每个操作都有一个明确定义的前驱和后继。这基本上是我们非常努力保留的编程模型。

传统模型是：单个程序，一个进程，一个内存空间运行在一个 CPU 上。操作系统抽象了可能有多個 CPU 和多個程序的事实，计算机上的内存实际上是在许多程序之间共享的。我并不是说线程编程和面向事件的编程不存在；只是它们是"一/一/一"模型之上的特殊抽象。程序的编写是为了按顺序执行：你从顶部开始，然后向下走向底部。

顺序作为一个属性受到如此多的关注，因为定义"正确性"的最简单方法是说"它像在单台机器上一样工作"。这通常意味着 a）我们运行相同的操作，b）我们按相同的顺序运行它们——即使有多台机器。

保留顺序（如单系统所定义）的分布式系统的好处在于它们是通用的。你不需要关心操作是什么，因为它们将完全像在单台机器上一样执行。这很棒，因为你知道无论操作是什么，你都可以使用相同的系统。

实际上，分布式程序运行在多个节点上；有多个 CPU 和多个操作流进入。你仍然可以分配一个总顺序，但这需要准确的时钟或某种形式的通信。你可以使用完全准确的时钟为每个操作打时间戳，然后使用该时钟找出总顺序。或者你可能有某种通信系统，使得可以按总顺序分配顺序号。

## 总顺序和偏序

分布式系统中的自然状态是 [偏序](https://en.wikipedia.org/wiki/Partially_ordered_set)。网络和独立节点都不保证相对顺序；但在每个节点上，你可以观察到一个局部顺序。

[总顺序](https://en.wikipedia.org/wiki/Total_order) 是一个二元关系，为某个集合中的每个元素定义一个顺序。

当两个不同元素中的一个大于另一个时，它们是**可比较的**。在偏序集中，某些元素对不可比较，因此偏序不指定每个项目的确切顺序。

总顺序和偏序都是 [传递的](https://en.wikipedia.org/wiki/Transitive_relation) 和 [反对称的](https://en.wikipedia.org/wiki/Antisymmetric_relation)。以下陈述在 X 中的所有 a、b 和 c 的总顺序和偏序中都成立：

如果 a ≤ b 且 b ≤ a，则 a = b（反对称性）；
如果 a ≤ b 且 b ≤ c，则 a ≤ c（传递性）；
然而，总顺序是 [完全的](https://en.wikipedia.org/wiki/Total_relation)：

对于 X 中的所有 a、b，a ≤ b 或 b ≤ a（完全性）

而偏序只是 [自反的](https://en.wikipedia.org/wiki/Reflexive_relation)：

对于所有 X 中的 a，a ≤ a（自反性）

注意，完全性隐含自反性；因此偏序是总顺序的较弱变体。对于偏序中的某些元素，完全性属性不成立——换句话说，某些元素不可比较。

Git 分支是偏序的一个例子。正如你可能知道的，git 版本控制系统允许你从单个基础分支（例如 master 分支）创建多个分支。每个分支代表基于共同祖先派生的源代码更改历史：

```
[ branch A (1,2,0)]  [ master (3,0,0) ]  [ branch B (1,0,2) ]
[ branch A (1,1,0)]  [ master (2,0,0) ]  [ branch B (1,0,1) ]
                  \  [ master (1,0,0) ]  /
```

分支 A 和 B 派生自共同祖先，但它们之间没有明确的顺序：它们代表不同的历史，不能简化为单个线性历史，除非进行额外的工作（合并）。当然，你可以将所有提交按某种任意顺序排列（例如，先按祖先排序，然后通过将 A 排在 B 之前或 B 排在 A 之前来打破平局）——但这会通过强制不存在的总顺序而丢失信息。

在由一个节点组成的系统中，总顺序必然出现：指令按特定的可观察顺序执行，消息在单个程序中处理。我们已经依赖这个总顺序——它使程序的执行可预测。这个顺序可以在分布式系统上维护，但有代价：通信昂贵，时间同步困难且脆弱。

## 什么是时间？

时间是顺序的来源——它允许我们定义操作的顺序——巧合的是，它也有一个人们可以理解的解释（一秒、一分钟、一天等等）。

在某种意义上，时间就像任何其他整数计数器一样。它恰好足够重要，以至于大多数计算机都有一个专用的时间传感器，也称为时钟。它如此重要，以至于我们已经想出了一些不完美的物理系统（从蜡蜡烛到铯原子）来合成相同计数器的近似值。通过"合成"，我的意思是，我们可以通过某种物理属性在物理上遥远的位置近似整数计数器的值，而无需直接通信。

时间戳实际上是一个简写值，表示从宇宙开始到当前时刻的世界状态——如果某事发生在特定的时间戳，那么它可能受到之前发生的一切的影响。这个想法可以推广到因果时钟，它明确跟踪原因（依赖关系），而不是简单地假设时间戳之前的一切都是相关的。当然，通常的假设是，我们应该只担心特定系统的状态，而不是整个世界。

假设时间在各处以相同的速率进步——这是一个很大的假设，我稍后会回到这个问题——时间和时间戳在用于程序时有几个有用的解释。这三种解释是：

*   **顺序**
*   **持续时间**
*   **解释**

**顺序**。当我说时间是顺序的来源时，我的意思是：

*   我们可以将时间戳附加到无序事件上以对它们进行排序
*   我们可以使用时间戳来强制执行特定的操作顺序或消息传递（例如，通过如果操作乱序到达则延迟它）
*   我们可以使用时间戳的值来确定某事是否在时间上发生在另一事之前

**解释**——时间作为普遍可比较的值。时间戳的绝对值可以解释为日期，这对人很有用。给定日志文件中停机开始的时间戳，你可以告诉那是上周六，当时有 [雷暴](https://twitter.com/AWSFail/statuses/218915147060752384)。

**持续时间**——用时间测量的持续时间与现实世界有一些关系。算法通常不关心时钟的绝对值或其作为日期的解释，但它们可能使用持续时间来做出一些判断。特别是，花费的等待时间可以提供关于系统是分区还是仅仅经历高延迟的线索。

就其本质而言，分布式系统的组件不以可预测的方式行为。它们不保证任何特定的顺序、进步速率或缺乏延迟。每个节点确实有一些局部顺序——因为执行（大致）是顺序的——但这些局部顺序彼此独立。

强加（或假设）顺序是减少可能执行和可能发生的空间的一种方法。当事物可以按任何顺序发生时，人类很难推理——有太多排列需要考虑。

## 时间在各处以相同的速率进步吗？

我们都有一个基于我们作为个人的经验的直观时间概念。不幸的是，这种直观的时间概念使得更容易想象总顺序而不是偏序。想象事物一个接一个地发生比同时发生更容易。推理单一的消息顺序比推理消息以不同的顺序和不同的延迟到达更容易。

然而，在实现分布式系统时，我们希望避免对时间和顺序做出强烈的假设，因为假设越强，系统对"时间传感器"——或板载时钟——的问题就越脆弱。此外，强加顺序是有代价的。我们可以容忍的时间非确定性越多，我们就越能利用分布式计算。

对于"时间在各处以相同的速率进步吗？"这个问题有三个常见的答案。这些是：

*   "全局时钟"：是
*   "局部时钟"：不，但是
*   "无时钟"：不！

这些大致对应于我在第二章中提到的三个定时假设：同步系统模型有一个全局时钟，部分同步模型有一个局部时钟，在异步系统模型中根本不能使用时钟。让我们更详细地看看这些。

### 带有"全局时钟"假设的时间

全局时钟假设是存在一个完美准确性的全局时钟，每个人都可以访问该时钟。这是我们倾向于思考时间的方式，因为在人类交互中，时间的微小差异并不重要。

![全局时钟](https://book.mixu.net/distsys/images/global-clock.png)

全局时钟基本上是总顺序的来源（即使这些节点从未通信，所有节点上每个操作的确切顺序）。

然而，这是对世界的理想化观点：实际上，时钟同步只能达到有限的准确性。这受到商品计算机中时钟缺乏准确性的限制，如果使用 [NTP](https://en.wikipedia.org/wiki/Network_Time_Protocol) 等时钟同步协议则受延迟限制，从根本上说受 [时空的本质](https://en.wikipedia.org/wiki/Time_dilation) 限制。

假设分布式节点上的时钟完全同步意味着假设时钟从相同的值开始并且永远不会漂移。这是一个很好的假设，因为你可以自由使用时间戳来确定全局总顺序——受时钟漂移而不是延迟限制——但这是一个 [非平凡的](http://queue.acm.org/detail.cfm?id=1773943) 操作挑战和潜在的异常源。有许多不同的场景，一个简单的故障——例如用户意外更改机器上的本地时间，或过时的机器加入集群，或同步的时钟以略微不同的速率漂移等等——可能导致难以追踪的异常。

尽管如此，确实有一些现实世界的系统做出这个假设。Facebook 的 [Cassandra](https://en.wikipedia.org/wiki/Apache_Cassandra) 是一个假设时钟同步的系统示例。它使用时间戳来解决写入之间的冲突——具有较新时间戳的写入获胜。这意味着如果时钟漂移，新数据可能被旧数据忽略或覆盖；同样，这是一个操作挑战（据我所知，人们敏锐地意识到这一点）。另一个有趣的例子是 Google 的 [Spanner](https://research.google.com/archive/spanner.html)：论文描述了他们的 TrueTime API，它同步时间但也估计最坏情况的时钟漂移。

### 带有"局部时钟"假设的时间

第二个，也许更合理的假设是每台机器都有自己的时钟，但没有全局时钟。这意味着你不能使用本地时钟来确定远程时间戳是否发生在本地时间戳之前或之后；换句话说，你不能有意义地比较来自两台不同机器的时间戳。

![局部时钟](https://book.mixu.net/distsys/images/local-clock.png)

局部时钟假设更接近现实世界。它分配一个偏序：每个系统上的事件是有序的，但不能仅使用时钟跨系统排序事件。

然而，你可以使用时间戳在单台机器上排序事件；只要小心不让时钟跳来跳去，你可以在单台机器上使用超时。当然，在由最终用户控制的机器上，这可能是假设太多：例如，用户在使用操作系统的日期控件查找日期时可能会意外地将他们的日期更改为不同的值。

### 带有"无时钟"假设的时间

最后，有逻辑时间的概念。在这里，我们根本不使用时钟，而是以其他方式跟踪因果关系。记住，时间戳只是到该点为止的世界状态的简写——所以我们可以使用计数器和通信来确定某事是发生在另一事之前、之后还是同时。

这样，我们可以确定不同机器之间事件的顺序，但不能说任何关于间隔的事情，也不能使用超时（因为我们假设没有"时间传感器"）。这是一个偏序：事件可以在单个系统上使用计数器排序而无需通信，但跨系统排序事件需要消息交换。

分布式系统中最常被引用的论文之一是 Lamport 关于 [时间、时钟和事件排序](http://research.microsoft.com/users/lamport/pubs/time-clocks.pdf) 的论文。向量时钟是该概念的推广（我将更详细地介绍），是一种不使用时钟跟踪因果关系的方法。Cassandra 的表亲 Riak (Basho) 和 Voldemort (Linkedin) 使用向量时钟，而不是假设节点可以访问完美准确性的全局时钟。这允许这些系统避免前面提到的时钟准确性问题。

当不使用时钟时，可以排序遥远机器上事件的最大精度受通信延迟限制。

## 时间在分布式系统中如何使用？

时间的好处是什么？

1.  时间可以定义跨系统的顺序（无需通信）
2.  时间可以定义算法的边界条件

事件顺序在分布式系统中很重要，因为分布式系统的许多属性是根据操作/事件的顺序定义的：

*   正确性取决于（同意）正确的事件排序，例如分布式数据库中的可串行性
*   当发生资源争用时，顺序可以用作决胜局，例如，如果有两个小部件的订单，履行第一个并取消第二个

全局时钟将允许在两台不同机器上的操作进行排序，而无需这两台机器直接通信。没有全局时钟，我们需要通信以确定顺序。

时间也可用于定义算法的边界条件——具体来说，区分"高延迟"和"服务器或网络链路宕机"。这是一个非常重要的用例；在大多数现实世界的系统中，超时用于确定远程机器是否发生故障，或者它是否只是经历高网络延迟。做出这种决定的算法称为故障检测器；我很快就会讨论它们。

## 向量时钟（因果顺序的时间）

早些时候，我们讨论了关于分布式系统中时间进步速率的不同假设。假设我们无法实现准确的时钟同步——或者从我们的系统不应该对时间同步问题敏感的目标开始，我们如何排序事物？

Lamport 时钟和向量时钟是物理时钟的替代品，它们依赖计数器和通信来确定分布式系统中事件的顺序。这些时钟提供一个可在不同节点之间比较的计数器。

**Lamport 时钟** 很简单。每个进程使用以下规则维护一个计数器：

*   每当进程做工作时，递增计数器
*   每当进程发送消息时，包含计数器
*   当收到消息时，将计数器设置为 `max(local_counter, received_counter) + 1`

用代码表示：

```javascript
function LamportClock() {
  this.value = 1;
}

LamportClock.prototype.get = function() {
  return this.value;
}

LamportClock.prototype.increment = function() {
  this.value++
}

LamportClock.prototype.merge = function(other) {
  this.value = Math.max(this.value, other.value) + 1;
}
```

[Lamport 时钟](https://en.wikipedia.org/wiki/Lamport_timestamps) 允许跨系统比较计数器，但有一个警告：Lamport 时钟定义一个偏序。如果 `timestamp(a) < timestamp(b)`：

*   `a` 可能发生在 `b` 之前，或者
*   `a` 可能与 `b` 不可比较

这称为时钟一致性条件：如果一个事件发生在另一个事件之前，那么该事件的逻辑时钟发生在其他事件之前。如果 `a` 和 `b` 来自相同的因果历史，例如，两个时间戳值都在同一进程上产生；或者 `b` 是对 `a` 中发送的消息的响应，那么我们知道 `a` 发生在 `b` 之前。

直观地说，这是因为 Lamport 时钟只能携带一个时间线/历史的信息；因此，比较从未相互通信的系统中的 Lamport 时间戳可能导致并发事件看起来是有序的，而实际上它们不是。

想象一个系统在初始阶段后分成两个独立的子系统，它们从不相互通信。

对于每个独立系统中的所有事件，如果 a 发生在 b 之前，则 `ts(a) < ts(b)`；但如果你从不同的独立系统中取两个事件（例如，不是因果相关的事件），那么你不能对它们的相对顺序说任何有意义的话。虽然系统的每个部分都已为事件分配了时间戳，但这些时间戳彼此没有关系。两个事件可能看起来是有序的，即使它们不相关。

然而——这仍然是一个有用的属性——从单台机器的角度来看，任何以 `ts(a)` 发送的消息都将收到一个响应 `ts(b)`，它是 `> ts(a)`。

**向量时钟** 是 Lamport 时钟的扩展，它维护一个包含 N 个逻辑时钟的数组 `[ t1, t2, ... ]`——每个节点一个。与其递增一个公共计数器，不如每个节点在每个内部事件上将其在向量中的自己的逻辑时钟递增一。因此更新规则是：

*   每当进程做工作时，递增向量中节点的逻辑时钟值
*   每当进程发送消息时，包含完整的逻辑时钟向量
*   当收到消息时：
    *   将向量中的每个元素更新为 `max(local, received)`
    *   递增向量中表示当前节点的逻辑时钟值

同样，用代码表示：

```javascript
function VectorClock(value) {
  // 表示为以节点 id 为键的哈希：例如 { node1: 1, node2: 3 }
  this.value = value || {};
}

VectorClock.prototype.get = function() {
  return this.value;
};

VectorClock.prototype.increment = function(nodeId) {
  if(typeof this.value[nodeId] == 'undefined') {
    this.value[nodeId] = 1;
  } else {
    this.value[nodeId]++;
  }
};

VectorClock.prototype.merge = function(other) {
  var result = {}, last,
      a = this.value,
      b = other.value;
  // 这过滤掉哈希中的重复键
  (Object.keys(a)
    .concat(b))
    .sort()
    .filter(function(key) {
      var isDuplicate = (key == last);
      last = key;
      return !isDuplicate;
    }).forEach(function(key) {
      result[key] = Math.max(a[key] || 0, b[key] || 0);
    });
  this.value = result;
};
```

这个插图（[来源](https://en.wikipedia.org/wiki/Vector_clock)）显示了一个向量时钟：

![向量时钟](https://book.mixu.net/distsys/images/vector_clock.svg.png)

三个节点（A、B、C）中的每一个都跟踪向量时钟。当事件发生时，它们用向量时钟的当前值打时间戳。检查向量时钟（如 `{ A: 2, B: 4, C: 1 }`）使我们能够准确识别（可能）影响该事件的消息。

向量时钟的问题主要在于它们需要每个节点一个条目，这意味着对于大型系统，它们可能会变得非常大。已经应用了各种技术来减少向量时钟的大小（要么通过执行定期垃圾回收，要么通过限制大小来降低准确性）。

我们已经研究了如何在没有物理时钟的情况下跟踪顺序和因果关系。现在，让我们看看时间持续时间如何用于截止。

## 故障检测器（截止的时间）

正如我早些时候所说的，花费的等待时间可以提供关于系统是分区还是仅仅经历高延迟的线索。在这种情况下，我们不需要假设完美准确性的全局时钟——仅仅有一个足够可靠的局部时钟就足够了。

给定一个运行在一个节点上的程序，它如何知道远程节点发生了故障？在缺乏准确信息的情况下，我们可以推断，在合理的时间内没有响应的远程节点已经发生故障。

但什么是"合理的数量"？这取决于本地和远程节点之间的延迟。与其用特定值明确指定算法（这在某些情况下必然是错误的），不如处理合适的抽象。

故障检测器是一种抽象掉确切定时假设的方法。故障检测器使用心跳消息和计时器实现。进程交换心跳消息。如果在超时发生之前未收到消息响应，则进程怀疑另一个进程。

基于超时的故障检测器将承担过于激进（声明节点发生故障）或过于保守（花费很长时间检测崩溃）的风险。故障检测器需要多准确才能可用？

[Chandra 等人](https://www.google.com/search?q=Unreliable%20Failure%20Detectors%20for%20Reliable%20Distributed%20Systems) (1996) 在解决共识的背景下讨论故障检测器——这个问题特别相关，因为它 underlying 大多数复制问题，其中副本需要在有延迟和网络分区的环境中达成一致。

他们使用两个属性来表征故障检测器：完整性和准确性：

| **强完整性** | 每个崩溃的进程最终被每个正确的进程怀疑 |
|------------|--------------------------------------|
| **弱完整性** | 每个崩溃的进程最终被某个正确的进程怀疑 |
| **强准确性** | 没有正确的进程被怀疑 |
| **弱准确性** | 某些正确的进程从不被怀疑 |

完整性比准确性更容易实现；实际上，所有重要的故障检测器都实现了它——你所需要做的就是不要永远等待怀疑某人。Chandra 等人指出，具有弱完整性的故障检测器可以转换为具有强完整性的故障检测器（通过广播关于可疑进程的信息），使我们能够专注于准确性属性的谱系。

避免错误地怀疑非故障进程是困难的，除非你能够假设消息延迟有一个硬性最大值。这个假设可以在同步系统模型中做出——因此故障检测器在这样的系统中可以是强准确的。在不强加消息延迟硬性边界的系统模型下，故障检测最多只能是最终准确的。

Chandra 等人表明，即使是一个非常弱的故障检测器——最终弱故障检测器 ⋄W（最终弱准确性 + 弱完整性）——也可用于解决共识问题。下图（来自论文）说明了系统模型和问题可解性之间的关系：

![来自 Chandra 和 Toueg. Unreliable failure detectors for reliable distributed systems. JACM 43(2):225–267, 1996.](https://book.mixu.net/distsys/images/chandra_failure_detectors.png)

正如你在上面看到的，某些问题在没有故障检测器的异步系统中是不可解的。这是因为没有故障检测器（或关于时间边界的强假设，例如同步系统模型），无法判断远程节点是崩溃了，还是只是经历高延迟。这个区别对于任何旨在实现单副本一致性的系统都很重要：失败的节点可以被忽略，因为它们不会导致发散，但分区的节点不能被安全地忽略。

如何实现故障检测器？从概念上讲，简单的故障检测器没有什么多的，它只是在超时到期时检测故障。最有趣的部分涉及如何做出关于远程节点是否发生故障的判断。

理想情况下，我们更喜欢故障检测器能够适应不断变化的网络条件，并避免将超时值硬编码到其中。例如，Cassandra 使用 [累进故障检测器](https://www.google.com/search?q=The+Phi+accrual+failure+detector)，它是一个输出怀疑级别（0 到 1 之间的值）而不是二元"向上"或"向下"判断的故障检测器。这允许使用故障检测器的应用程序自己对准确检测和早期检测之间的权衡做出决定。

## 时间、顺序和性能

早些时候，我暗示必须为顺序付出代价。我是什么意思？

如果你正在编写分布式系统，你大概拥有多台计算机。世界的自然（和现实）观点是偏序，而不是总顺序。你可以将偏序转换为总顺序，但这需要通信、等待并施加限制，限制在任何特定时间点有多少计算机可以做工作。

所有时钟都只是受网络延迟（逻辑时间）或物理限制的近似值。即使在多个节点之间保持一个简单的整数计数器同步也是一个挑战。

虽然时间和顺序经常一起讨论，但时间本身并不是这样一个有用的属性。算法并不像关心更抽象的属性那样关心时间：

*   事件的因果顺序
*   故障检测（例如，消息传递上限的近似值）
*   一致的快照（例如，在某个时间点检查系统状态的能力；这里不讨论）

强加总顺序是可能的，但很昂贵。它需要你以共同（最低）速度前进。通常，确保事件按某些定义的顺序传递的最简单方法是指定一个单个（瓶颈）节点，所有操作都通过该节点传递。

时间/顺序/同步真的必要吗？这取决于。在某些用例中，我们希望每个中间操作将系统从一个一致状态移动到另一个一致状态。例如，在许多情况下，我们希望来自数据库的响应表示所有可用信息，我们希望避免处理如果系统返回不一致结果可能发生的问题。

但在其他情况下，我们可能不需要那么多时间/顺序/同步。例如，如果你正在运行一个长时间运行的计算，并且直到最后才真正关心系统做什么——那么你不需要太多同步，只要你能保证答案是正确的。

同步通常作为对所有操作的 blunt 工具应用，而只有一部分情况实际上对最终结果很重要。什么时候需要顺序来保证正确性？CALM 定理——我将在最后一章讨论——提供了一个答案。

在其他情况下，给出一个只表示最佳已知估计的答案是可以接受的——也就是说，仅基于系统中包含的总信息的子集。特别是，在网络分区期间，可能只能使用系统的一部分来回答查询。在其他用例中，最终用户无法真正区分可以廉价获得的相对较新的答案和保证正确但计算昂贵的答案。例如，某些用户 X 的 Twitter 粉丝计数是 X 还是 X+1？或者电影 A、B 和 C 是某个查询的绝对最佳答案吗？做更便宜的、大部分正确的"尽力而为"是可以接受的。

在接下来的两章中，我们将检查用于容错强一致性系统的复制——这些系统在提供强保证的同时对故障越来越有弹性。这些系统为第一种情况提供解决方案：当你需要保证正确性并愿意为此付出代价时。然后，我们将讨论具有弱一致性保证的系统，这些系统在面临分区时可以保持可用，但只能给你一个"尽力而为"的答案。
