Netflix Live Origin 是自定义构建的服务器，位于云直播流式传输管道和 Open Connect（Netflix 的内容分发网络 CDN）之间。它充当质量控制检查点，决定哪些视频段交付给全球数百万观众。

当 Netflix 首次引入直播流式传输时，他们需要一个可以处理实时视频交付独特挑战的系统。与视频点播（VOD）不同（内容提前准备），直播流式传输在时间约束下运行。每个视频段必须在几秒内编码、打包和交付给观众。Live Origin 专门设计用于处理这些需求。

在本文中，我们将看看这个系统的架构以及 Netflix 在构建它时面临的挑战。

免责声明：这篇文章基于 Netflix 工程团队公开分享的细节。如果你发现任何不准确之处，请评论。

## 架构

Live Origin 作为 Amazon EC2 实例上的多租户微服务运行。通信模型很简单：

- **Packager**（准备视频段进行分发）使用 HTTP PUT 请求将这些段发送到 Origin
- **Open Connect 节点**使用 HTTP GET 请求检索这些段。用于 PUT 请求的 URL 与用于 GET 请求的 URL 匹配，创建简单的存储和检索模式

Netflix 做了一些架构决策，塑造了 Live Origin 的功能。

### 1. 冗余区域直播流式传输管道

他们通过冗余区域直播流式传输管道实现可靠性。不是依赖单个编码管道，Netflix 同时运行两个独立的管道。每个管道在不同云区域中运行，有自己的编码器、packager 和视频贡献 feed。

### 2. 段模板和恒定段持续时间

Netflix 采用了带有段模板和恒定段持续时间的清单设计。不是不断更新清单文件列出可用段，他们使用可预测模板，每个段有固定的 2 秒持续时间。这个设计选择允许 Origin 精确预测段何时应该发布。

## 缺陷处理

直播视频流不可避免地包含缺陷，因为直播视频 feed 的不可预测性和严格的实时发布截止时间。常见问题包括缺少视频帧或音频样本的短段、完全缺失的段，以及解码时间戳不正确的时序不连续。

运行两个独立管道大大降低了两个管道同时产生缺陷段的概率。因为管道使用不同的云区域、编码器和视频源，当一个管道产生坏段时，另一个通常产生好的。

Live Origin 利用其在分发路径中的位置做出智能决策。当 Open Connect 请求段时，Origin 按预定顺序检查来自两个管道的候选者，并选择第一个有效的。为了检测缺陷，Packager 执行轻量级媒体检查，并在将段发布到 Origin 时将缺陷信息作为元数据包含。在两个管道都有缺陷段的罕见情况下，这个信息传递到下游，以便客户端可以相应地处理错误。

## Open Connect 优化

当 Live 项目启动时，Open Connect 高度优化用于 VOD 交付。Netflix 花了多年时间调整 nginx（Web 服务器）和底层操作系统用于这个用例。与传统 CDN 按需填充缓存不同，Open Connect 预先将 VOD 资产放置在精心选择的服务器上，称为 Open Connect Appliances（OCA）。

直播流式传输不适合这个模型，所以 Netflix 扩展了 nginx 的代理缓存功能。他们做了一些优化来减少不必要的流量并提高性能。

### 1. 请求过滤

Open Connect 节点接收与客户端提供的相同段模板。使用模板信息，OCA 可以确定任何事件在任何时间点的段合法范围。这允许它们立即拒绝此范围外的段请求，防止不必要的请求通过网络传播到 Origin。

### 2. 404 缓存

当段还不可用且 Origin 收到请求时，Origin 返回 404 状态码（文件未找到）以及过期策略。Open Connect 可以缓存这个 404 响应直到段预计发布前。这防止重复的失败请求。

### 3. 实时边缘优化

Netflix 为实时边缘的请求实现了一个巧妙的优化。当请求下一个即将发布的段时，Origin 保持请求打开。一旦段发布，Origin 立即响应等待的请求。这显著减少了网络中略早到达请求的流量。

### 4. 毫秒级缓存控制

为了支持这个功能，Netflix 向 nginx 添加了毫秒级缓存控制。标准 HTTP 缓存控制只在秒级工作，当段每 2 秒生成时太粗糙。

## 通知系统

Netflix 使用自定义 HTTP 头以高度可扩展的方式传达流式传输事件。直播流式传输管道向 Origin 提供通知，Origin 将它们作为 HTTP 头插入在该点生成的段上。这些头是累积的，意味着它们持续到未来的段。

每当段到达 OCA 时，通知信息从响应头中提取并使用事件 ID 作为键存储在内存中。当 OCA 向客户端提供段时，它将最新通知数据附加到响应。这个系统确保无论观众在流中的位置如何，他们都接收最新的通知数据。通知甚至可以在不提供新段的响应上传达。

这个方法允许 Netflix 有效地向数百万设备传达信息，如广告休息、内容警告或直播事件更新，独立于他们的播放位置。

## 存储架构

Netflix 最初使用 AWS S3 用于 Live Origin 存储，类似于他们的 VOD 基础设施。这对低流量事件工作良好，但随着他们扩展，他们发现直播流式传输有与按需内容显著不同的独特需求。

虽然 S3 满足其声明的正常运行时间保证，但直播事件固有的严格 2 秒重试预算意味着任何延迟都有问题。在直播流式传输中，每次写入都至关重要且时间敏感。需求更接近全球、低延迟、高可用数据库，而不是对象存储。

Netflix 为新存储系统建立了五个关键需求：

1. **极高写入可用性**：在单个 AWS 区域内，低延迟复制到其他区域。500 毫秒内的任何失败写入都被视为 bug
2. **高写入吞吐量**：处理数百兆字节跨区域复制
3. **大写入支持**：有效支持累积到每个分区数千个键的大写入
4. **强一致性**：在同一区域内需要强一致性以实现低于 1 秒的读取延迟
5. **高读取吞吐量**：在最坏情况下可能需要处理千兆字节级的读取吞吐量而不影响写入

## 解决方案

Netflix 之前构建了一个键值存储抽象，利用 Apache Cassandra 提供大值的分块存储。这个抽象最初构建用于支持云游戏保存，但 Live 用例将在写入可用性、累积分区大小和读取吞吐量方面推动其边界。

解决方案将大负载分解为可以独立重试的块。结合 Apache Cassandra 的本地法定人数一致性模型（即使整个可用区中断也允许写入可用性）和写入优化的日志结构合并树存储引擎，Netflix 可以满足前四个需求。

性能改进很稳固。中位延迟从 113 毫秒下降到 25 毫秒，第 99 百分位延迟从 267 毫秒提高到 129 毫秒。这个新解决方案更昂贵，但最小化成本不是主要目标。

然而，处理 Origin Storm 失败情况需要额外工作。在这个场景中，数十个 Open Connect 顶级缓存可以同时请求多个大视频段。计算显示最坏情况读取吞吐量可能达到 100 Gbps 或更多。

Netflix 可以从 Apache Cassandra 以网络线速率响应读取，但观察到并发写入的性能下降不可接受。为了解决这个问题，他们使用 EVCache（基于 Memcached 的分布式缓存系统）引入了写透缓存。这允许几乎所有读取从高度可扩展的缓存服务，实现 200 Gbps 及以上的吞吐量而不影响写入路径。

在最终架构中，Live Origin 写入和读取 KeyValue，它管理到 EVCache 的写透缓存，并实现分块协议，将大值分散到 Apache Cassandra 存储集群。几乎所有读取负载从缓存处理，只有缓存未命中击中存储层。这个组合成功满足 Live Origin 的苛刻需求超过一年。

## 规模和流量管理

Netflix 的直播流式传输平台为每个直播事件处理大量多样化的流渲染。这种复杂性来自支持各种视频编码格式与多个质量级别、多种语言和格式的音频选项，以及不同内容版本（如有或没有广告）。在 2024 年 Tyson vs. Paul 比赛事件期间，Netflix 观察到 6500 万并发流的历史峰值。

Netflix 选择构建高度可扩展的 origin 而不是依赖传统 origin shield 以获得更好的缓存一致性控制和更简单的系统架构。Live Origin 直接连接到地理分布在不同站点的顶级 Open Connect 节点。为了最小化 origin 的负载，每个流渲染在每个站点只有指定节点可以直接从 origin 填充。

虽然 origin 服务可以使用 EC2 实例水平扩展，但其他系统资源（如存储平台容量和骨干带宽）不能自动扩展。不是所有对 live origin 的请求都有相同的重要性。Origin 写入最关键，因为失败直接影响流式传输管道。实时边缘的 Origin 读取最关键，因为失败影响大多数客户端。DVR 模式的 Origin 读取不太关键，因为失败只影响倒带的客户端。

Netflix 实施全面的发布隔离来保护对延迟敏感和故障敏感的 origin 写入。origin 为发布和 CDN 流量使用单独的 EC2 栈。存储抽象层为读取和写入操作具有不同的集群。存储层本身通过 EVCache 将读取路径与通过 Cassandra 的写入路径分开。这个完整的路径隔离允许发布和检索流量独立扩展，防止 CDN 流量激增影响发布性能。

当底层系统经历压力时，Live Origin 实施基于优先级的速率限制。这个方法确保在负载高峰期对用户体验影响更大的请求成功，而影响较小的请求允许失败。在存储平台负载高峰期，实时边缘流量优先于 DVR 流量。检测基于可预测的段模板，它在 origin 节点内存中缓存。这允许在数据存储之前做出优先级决策，在数据存储压力期间特别有价值。

为了减轻流量激增，Netflix 与优先级速率限制一起使用 TTL 缓存控制。当低优先级流量受影响时，Origin 指示 Open Connect 通过设置 max-age 指令并返回 HTTP 503 错误码来缓存相同请求 5 秒。这个策略通过防止 5 秒窗口内的重复请求来抑制流量激增。

发布隔离和优先级速率限制成功保护 live origin 免受 DVR 流量风暴。然而，由不存在段请求生成的流量风暴提出了额外的挑战和优化机会。

## 元数据缓存

Live Origin 将元数据分层组织为事件、流渲染和段。段发布模板在流渲染级别维护。这个分层组织允许 origin 使用高度可缓存的事件和流渲染级别元数据预先拒绝请求，避免不必要的段级别元数据查询。

流程如下：

- 如果事件未知，请求用 404 错误拒绝
- 如果事件已知但段请求时间与预期发布时间不匹配，请求用 404 拒绝，缓存控制 TTL 与预期发布时间匹配
- 如果事件已知但请求的段从未生成或错过重试截止时间，请求用 410（Gone）错误拒绝，告诉客户端停止请求

在存储层，元数据与控制平面数据存储中的媒体数据分开存储。与媒体数据存储不同，控制平面数据存储不使用分布式缓存以避免缓存不一致。事件和渲染级别元数据受益于使用内存缓存时的高缓存命中率。在涉及不存在段的流量风暴期间，控制平面访问的缓存命中率轻松超过 90%。

元数据的内存缓存有效处理 live origin 的 404 风暴而不造成数据存储压力。这个元数据缓存补充了存储系统的分布式媒体缓存，为流量激增保护提供完整解决方案。

## 结果

Netflix Live Origin 是专门为大规模直播流式传输的独特需求构建的复杂系统。

通过冗余管道、智能段选择、优化缓存策略、优先级请求处理和自定义存储架构，Netflix 创建了一个可靠的基础，用于向全球数百万并发观众交付直播事件。

该系统成功平衡了写入可靠性、读取可扩展性和运营灵活性的竞争需求，在 Tyson vs. Paul 比赛等重大事件期间证明了其有效性，达到 6500 万并发流。

## 译者总结

这篇文章介绍了 Netflix Live Origin 的架构：

**核心挑战**：
- 实时视频交付（2 秒段）
- 高并发（6500 万并发流）
- 低延迟要求

**关键架构决策**：
- 冗余区域管道（两个独立管道）
- 段模板和恒定持续时间
- 智能段选择（选择第一个有效段）

**优化策略**：
- 请求过滤和 404 缓存
- 实时边缘请求保持打开
- 毫秒级缓存控制
- 自定义 HTTP 头通知系统

**存储架构**：
- Apache Cassandra + EVCache
- 写透缓存
- 分块协议
- 读写路径隔离

**流量管理**：
- 基于优先级的速率限制
- TTL 缓存控制
- 元数据分层和内存缓存

**关键洞察**：
- 冗余管道减少缺陷概率
- 路径隔离允许独立扩展
- 元数据缓存有效处理 404 风暴
- 自定义存储满足直播独特需求

理解这些概念对构建大规模直播流式传输系统至关重要。
